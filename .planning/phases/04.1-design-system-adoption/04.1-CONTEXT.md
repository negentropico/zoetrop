# Phase 04.1: Design System Adoption - Context

**Gathered:** 2026-06-07
**Status:** Ready for planning

<domain>
## Phase Boundary

Retrofit the shipped M0 app into the realized **Zoetrop** brand and commit a binding design contract. Concretely: bridge the brand DS tokens into Tailwind 4 `@theme`, swap fonts (Inter → Space Grotesk / Hanken Grotesk / Space Mono), port the signature components from inline-style `.jsx` to typed Tailwind TSX under `app/components/ui/`, and reskin the app's routes in-brand. The visual design is already settled — a claude.ai/design roundtrip delivered a complete realized redesign (`docs/design-system/_rounds/round1/`) and resolved all open questions. The binding contract is `04.1-UI-SPEC.md` (already approved).

Out of scope: changing information architecture, routes, or data sources; new features; live-DB wiring (Phase 4); lab-ingest/report UI (Phases 5–6, which build *against* this contract).

</domain>

<decisions>
## Implementation Decisions

### Integration architecture (locked — DESIGN-SYSTEM-ADOPTION.md §6)
- **D-01:** Token bridge into Tailwind 4 `@theme` — map DS CSS-variable tokens to `@theme` tokens so the app keeps writing utilities (`bg-paper`, `text-ink`, `font-display`, `text-vital`, `rounded-frame`, `shadow-warm-md`) that now resolve to brand values. Re-skinning = class swaps, not a rewrite.
- **D-02:** Port signature components to **typed Tailwind TSX** under `app/components/ui/`. Do **not** ship the `.jsx` / `_ds_bundle.js` runtime into the TS-strict app.
- **D-03:** **No shadcn.** Tooling = none (Tailwind `@theme` + hand-ported TSX). `docs/design-system/` stays the source of truth; the app consumes a bridged subset. Document the token→Tailwind mapping so drift is visible.

### Resolved design questions (from roundtrip — see UI-SPEC + screen-about.jsx)
- **D-04 (Q1):** Color is **status-driven**, not category-driven. The 9 clinical categories get identity from a **Lucide icon in a neutral Ink chip** — no per-category hue. The 3 brand families (Energy/Vital/Focus) stay reserved for the Autonomic/activity domain (HRV→Vital, sleep→Focus, strain→Energy).
- **D-05 (Q1b):** Emoji → Lucide mapping per UI-SPEC Category System table (pill, gem, flame, zap, flask-conical, heart-pulse, dumbbell, droplet, dna).
- **D-06 (Q2):** Four-status palette — optimal=Vital teal, borderline=Energy amber, deficient=red, **excess=deep bronze `#c98910`** (add `--excess`/`--excess-bg` to the `@theme`; not in base `colors.css`). Each status pairs with a glyph (✓ ~ ↓ ↑) + mono label for colorblind safety.
- **D-07 (Q4):** One responsive system — desktop `TopNav` collapses to `BottomTab` (≥44px targets) at ~760px; Metrics section nav = horizontal pill scroll; multi-column grids reflow to 1 col; dense `DataTable` switches table→card-per-row on mobile.
- **D-08 (Q5):** Wordmark = `zoetrop.` (periwinkle period) + small `codename Zoetrop` tag. Flag, do not relitigate naming.

### Dark mode (decided this session — supersedes adoption-doc D3)
- **D-09:** **Ship dark mode in this phase.** The roundtrip delivered a full warm-dark token remap and a `ThemeToggle`; the UI-SPEC specifies it. Wire the `html[data-theme]` attribute, the no-flash inline theme script (reads `localStorage` `zt-theme`, falls back to `prefers-color-scheme`), the warm-dark token block, and the toggle. (Original D3 "defer dark" is explicitly superseded — note this in DESIGN-SYSTEM-ADOPTION.md.)

### Screen coverage (decided this session)
- **D-10:** **Retrofit all 16 routes**, not just the 8 archetypes. The 8 roundtrip-realized archetypes drive the component/pattern decisions; the remaining 8 routes (`protocol/versions`, `protocol/version-detail`, `protocol/supplements`, `protocol/compare`, `insights/index`, `insights/genetics`, `metrics/index`, `import/index`, `import/vault`) follow the same ported components and tokens. No half-branded app.

### Dependencies & assets
- **D-11:** Add `lucide-react` for icons (replaces emoji). Load the three brand fonts via Google Fonts `<link>` in `root.tsx` (replacing the Inter link), per the substitution flags. Keep these flagged as stand-ins for the licensed grotesque.

### Fidelity conflicts
- **D-12:** Where the brand readme and the round1 realized files disagree, the **round1 realized files win** (e.g. radii: use the round1 reduced set `--radius-xl: 14px`, not the readme's 28px). The UI-SPEC already encodes the realized values.

### Claude's Discretion
- Plan chunking / wave structure (e.g. foundations → components → screen retrofits) and PR-ability.
- Test approach for ported components (the vitest harness exists from Phase 1).
- Exact file/folder layout under `app/components/ui/` (and a `shell/` subfolder for TopNav/BottomTab if useful).
- Whether to keep/rename the existing `app/components/TrendChart.tsx` while restyling it to the brand `TrendChart`.

</decisions>

<specifics>
## Specific Ideas

- The realized redesign in `docs/design-system/_rounds/round1/` is the visual target — port its components and screen compositions faithfully (adapting hash-router/CDN patterns to React Router 7 + loaders).
- Preserve every number and the chart semantics (actual vs target vs optimal/reference bands) on metric detail — restyle to brand colors, keep the meaning.
- Density appropriate to a practitioner instrument: calm, precise — not a consumer toy.
