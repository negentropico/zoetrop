# Phase 4.1 — Design Critique: Zoetrope brand retrofit

> Produced by the `design-cognition` Critic role. Findings are specific, grounded,
> and prioritized by impact (Critical / Significant / Minor) with a resolution
> direction + trade-off for each. C1 and S5 were confirmed in the DOM + source,
> not by eye alone.

## Method

Rendered the **current** local build (`npm run dev`, port 5174) — the May-19
screenshots in `_notes/` show the superseded pre-brand UI and were **not**
critiqued. Coverage: Dashboard (desktop + mobile, light + dark), Metrics category,
Metric detail, Genetics, Correlations (desktop + mobile + DOM inspection), Import.
Dark theme tested at runtime **and** on persisted-preference reload.

## Strengths (verified, not manufactured)

- Brand retrofit is genuinely applied and consistent: warm Paper, Space
  Grotesk/Hanken/Space Mono, SpiralMark wordmark, frame cards, soft Ink-tinted
  shadows, zero gradients, zero emoji.
- **Status system is colorblind-redundant** — glyph + mono label + color on every
  status element.
- **Confidence is a first-class column** on Genetics — the right instinct for the
  product's core thesis.
- Dark-theme CSS is fully token-driven; the **runtime toggle is clean** (bg flips
  to `#15110f`, warm not blue-grey).
- Mono-for-data discipline (tabular nums, units) holds across screens.

This is a competent, on-contract build. The findings below are specific defects,
not a verdict on the whole.

## CRITICAL

### C1 — Correlations renders every row twice on desktop
`/insights/correlations` shows the dense `<table>` **and** the mobile
card-per-row list at the same time at all desktop widths. Confirmed in DOM:
viewport 1280, `+0.82` appears twice in body text; one visible `<table>` + a
visible `.corr-mobile` computing to `display:flex`.

- **Mechanism:** `remix-app/app/routes/insights/correlations.tsx:403` sets an
  inline `style={{ display: "flex", … }}` on the `.corr-mobile` div. Inline
  styles beat stylesheet rules, overriding `.corr-mobile { display: none }`
  (line 475) and the `@media (max-width:760px)` toggle (lines 476–479). The CSS
  toggle is correct; one inline `display` defeats it.
- **Fix direction:** remove `display: "flex"` from the inline style on line 403
  (keep `flexDirection`/`gap`); let the class + media query own `display`. The
  media block already restores `display:flex` for mobile.
- **Trade-off:** none — pure fix.
- *Invisible in static design review; only exists in the running build.*

## SIGNIFICANT

### S1 — Confidence vocabulary fragmented 3 ways (on the product's core screen)
`/insights/genetics` uses three non-aligned vocabularies for the one concept the
product exists to communicate ("confidence under genuine uncertainty"):
- Stat tiles: `K1 CONFIRMED · K2 HIGH · K3 INFERRED` (K-numbers, no K4)
- Table badges: `CONFIRMED · INFERRED` only (no K-number)
- Guide: `Confirmed · Likely · Inferred · Requires Testing` (4 words, no K-numbers)

A reader can't map badge-`INFERRED` → tile-`K3` → guide-`Inferred`, can't see
where `K2/Likely` went (zero rows), or what `K4/Requires Testing` is.
- **Fix direction:** one canonical 4-level scale (the K1–K4 system), render the
  K-number on **every** badge (`K1 · CONFIRMED`), and make tiles + guide use
  identical labels.
- **Trade-off:** slightly wider badges. *Lens: System + User.*

### S2 — Same value, two units on one page
`/metrics/.../homocysteine`: header readout is `9.40 MMOL/L`, but RangeBar,
Details (`5–15 µmol/L · optimal 5–8`), Targets, and the Import Phase-5 card all
say `µmol/L`. Homocysteine is µmol/L — the readout is wrong/inconsistent.
- **Fix direction:** source the unit from a single field; the `Stat`/readout
  component formats a different unit string than the detail panels.
- **Trade-off:** none. *Lens: Logic.* (Placeholder *ranges* are a documented
  substitution; the *inconsistency* is not.)

### S3 — TrendChart misleads at n=1
With `MEASUREMENTS 1`, the chart plots a single actual point (9.4 @ M2) beside two
**target** circles (8.0/7.0 @ M5/M6), shows an "ACTUAL" line legend but draws no
line, and the `9.8` y-tick collides with the `9.4` data label (point clipped at
card top). Reads as a downward trend that is actually just two future targets.
- **Fix direction:** require ≥2 actual points to draw the line; below that show a
  "Not enough data for a trend yet" state and render targets visually distinct
  from the actual series.
- **Trade-off:** blank-ish chart for sparse metrics, but honest. *Lens: Edge case + User.*

### S4 — Cessation tracker has no terminal/overrun state
Dashboard + `/protocol/cessation` show `Day 168 of 150`, all four PhaseBar
segments complete, yet copy still reads "Optimization phase — Tier 1 supplements
only" as if ongoing (168/150 = 112%).
- **Fix direction:** add a `complete` state (Day N · protocol complete, all phases
  filled, closing line) when `day > totalDays`.
- **Trade-off:** one more state; unavoidable since the real start date already
  crossed the line. *Lens: Edge case.*

### S5 — Dark mode doesn't survive a reload (persisted-preference path)
Runtime toggle works perfectly, but with `localStorage zt-theme="dark"`, a fresh
load leaves `data-theme` **null** and renders **light**, while ThemeToggle shows
the dark state (sun icon, `aria-label="Switch to light theme"`) — a visible
desync. The no-flash inline script is present and reads `'dark'`, but the
attribute it sets pre-paint is dropped during React hydration (light happens to
match the server default, so only **dark** breaks).
- **Fix direction (canonical RR7/Remix):** store theme in a **cookie**, read it in
  the root loader, server-render `<html data-theme={theme}>` — no
  inline-script-vs-hydration race. Cheaper patch: re-assert `data-theme` from
  `localStorage` in a root mount effect (still flashes).
- **Trade-off:** cookie adds a loader read; eliminates the bug class. *Lens: System + Edge case.*

### S6 — "Need a look: 3" carries a borderline signal while 15 are borderline
Dashboard tile shows `3` with a `~ BORDERLINE` badge (amber, borderline glyph),
but Metric status = Optimal 20 / Borderline 15 / Deficient 1 / Excess 2. The `3`
almost certainly = deficient + excess (out-of-range), yet it's tagged borderline,
and the 15 actually-borderline metrics get no surfaced count.
- **Fix direction:** decide what "need a look" means and label honestly — either
  "3 out of range (↓/↑)" or split into out-of-range vs borderline counts.
- **Trade-off:** slightly more tile content. *Lens: Logic + User.*

## KEY ASSUMPTION TO TEST (Critic → Strategist handoff)

Every Significant finding shares one root: **the build is a faithfully-polished
n=1 personal instrument, and the failure modes all live exactly where
single-subject assumptions meet real-world data** — one measurement (S3), one
person's protocol run past its end (S4), placeholder units/ranges (S2/M3), a
confidence taxonomy fine as private shorthand but fragmenting the moment it's
shown as a feature (S1).

> **Assumption to test:** that the right move now is to keep hardening the n=1
> instrument to this fidelity — rather than to validate the multi-tenant /
> engine-as-moat direction the platform brief (`docs/PLATFORM.md`) commits to.
>
> What would have to be true for "polish the personal dashboard" to be correct?
> - (a) The n=1 instrument **is** the demo that wins the HIGHER pilot → its polish
>   is the sales artifact → S1–S4 are **pilot-blocking** and fix before any demo.
> - (b) M1 multi-tenant is the real next bet → the confidence taxonomy (S1) and the
>   data-sparsity/overrun states (S3/S4) must be **designed for many subjects**,
>   not patched for one.
>
> The build doesn't tell us which. That's a Strategist question, and it changes
> what "fix" even means. Flagged, not answered (scope was Critic).
