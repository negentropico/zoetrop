## Summary

Phase 04.1 is an implementation phase, not a design phase вЂ” the visual design is fully resolved in `04.1-UI-SPEC.md` and `docs/design-system/_rounds/round1/`. The research question is: how do you land that settled design faithfully into a React Router 7 + TypeScript-strict + Tailwind CSS 4.1 codebase, and how do you verify it?

The central mechanic is **`@theme inline` with semantic CSS variable forwarding**. The DS token files (`colors.css`, `typography.css`, `spacing.css`) define `:root` CSS custom properties. The bridge imports those token files and re-declares the same values in `@theme inline` using the Tailwind 4 namespace rules (`--color-*`, `--font-*`, `--radius-*`, `--shadow-*`). Because `@theme inline` resolves variable references at compile-time, utilities like `bg-paper` generate with a static hex value вЂ” but the cascading overrides in `html[data-theme="dark"]` still win at runtime, because utilities that reference brand tokens (via `var(--paper)` in `@layer base`) automatically flip. The recommended architecture uses semantic tokens as the bridge layer: declare `--color-paper: #f5f2f0` in `@theme inline`, let the dark block remap `--paper`, and use `bg-[var(--paper)]` or a utility class in your component, not `bg-paper` directly if you need dynamic cascade.

The no-flash dark mode script and `suppressHydrationWarning` pattern for React Router 7 is well-established. The blocking inline `<script>` goes inside the `Layout` export's `<head>` using `dangerouslySetInnerHTML`, executed before first paint. The `<html>` element receives `suppressHydrationWarning` to suppress the server/client attribute mismatch. `lucide-react@1.17.0` is the current package (confirmed via npm registry), tree-shakes correctly with Vite, and is used as named imports (`import { Pill } from "lucide-react"`). Recharts 3 accepts `stroke="var(--ink)"` and similar CSS variable strings on all path/dot props вЂ” the chart themes via CSS variable cascade, no special Recharts API needed.

**Primary recommendation:** Structure the build as three waves: (1) Token bridge + fonts + base helpers in `app.css` and `root.tsx`; (2) Port all components to `app/components/ui/` and `app/components/shell/`; (3) Retrofit all 16 routes screen by screen.

## Tailwind CSS 4 `@theme` Token Bridge вЂ” Core Mechanics

### How `@theme` generates utilities

[VERIFIED: tailwindcss.com/docs/theme] In Tailwind CSS 4, variables declared inside `@theme` instruct the compiler to generate utility classes AND output CSS variables in `:root`. The namespace rules are:

| `@theme` variable prefix | Generated utility namespace | Example |
|--------------------------|----------------------------|---------|
| `--color-*` | `bg-*`, `text-*`, `border-*`, `fill-*`, `stroke-*` | `--color-paper` в†’ `bg-paper`, `text-paper` |
| `--font-*` | `font-*` | `--font-display` в†’ `font-display` |
| `--text-*` | `text-*` (font-size) | `--text-xl` в†’ `text-xl` |
| `--radius-*` | `rounded-*` | `--radius-xl` в†’ `rounded-xl` |
| `--shadow-*` | `shadow-*` | `--shadow-sm` в†’ `shadow-sm` |
| `--spacing-*` | All spacing utilities | `--spacing-*` в†’ `p-*`, `m-*`, `gap-*` |
| `--ease-*` | `ease-*` | `--ease-out` в†’ `ease-out` |

### `@theme` vs `@theme inline` вЂ” the critical distinction

[VERIFIED: tailwindcss.com/docs/theme + github.com/tailwindlabs/tailwindcss/discussions/15083]

- **`@theme { --color-paper: #f5f2f0; }`** вЂ” static hex value baked into utility class. The utility `bg-paper` always resolves to `#f5f2f0` regardless of runtime CSS var changes. Dark mode via `html[data-theme="dark"]` remapping `--paper` does NOT affect this utility.
- **`@theme inline { --color-paper: var(--paper); }`** вЂ” Tailwind emits `bg-paper { background-color: var(--paper); }`. The utility references the CSS variable at runtime. When `html[data-theme="dark"]` remaps `--paper: #15110f`, the `bg-paper` utility automatically picks up the dark value. **This is the correct pattern for this project.**

### The bridge architecture

The DS token files define `:root` custom properties using their own names (`--paper`, `--ink`, `--energy`, etc.). The bridge in `app.css` works in three layers:

```css

@import "../../../docs/design-system/tokens/typography.css";
@import "../../../docs/design-system/tokens/spacing.css";

/* Layer 2: add missing tokens not in DS files (from HTML :root override) */
:root {
  --excess: #c98910;
  --excess-bg: #f7edd8;
  /* reduced radii (round1 values override spacing.css) */
  --radius-xs: 2px;
  --radius-sm: 4px;
  --radius-md: 7px;

  --radius-xl: 14px;

Since `--font-display` is already in `:root` from the imported token file, utilities like `font-display` work correctly. **Custom `--font-*` naming:** Tailwind 4's `--font-*` prefix generates `font-{name}` utilities. Naming it `--font-display` generates `font-display`, and `--font-text` generates `font-text`. The `--font-sans` default is left in place (or cleared with `--font-sans: initial`) so Tailwind base styles keep working. [VERIFIED: tailwindcss.com/docs/theme]

### Dark mode variant setup

[VERIFIED: tailwindcss.com/docs/dark-mode]

```css
/* In app.css, after @import "tailwindcss" */
@custom-variant dark (&:where([data-theme=dark], [data-theme=dark] *));
```

This enables `dark:` utilities driven by `html[data-theme="dark"]`. Components can then use `dark:bg-black` syntax. However, for the semantic token system, the dark remap via `html[data-theme="dark"]` CSS overrides is the primary mechanism вЂ” `dark:` utilities are a secondary/supplement pattern for things the semantic cascade cannot cover (e.g. `dark:color-scheme: dark`).

### Dark token remap block

```css
html[data-theme="dark"] {
  color-scheme: dark;
  --paper: #15110f;
  --paper-2: #1e1917;
  --n-0: #221d1b;
  /* ... full table from UI-SPEC.md В§Dark Theme ... */

Because `@theme inline` makes utilities reference `var(--paper)` etc. at runtime, and the dark block remaps `--paper` on `html`, all brand utilities flip automatically with zero `dark:` prefixes in component JSX. [VERIFIED: github.com/tailwindlabs/tailwindcss/discussions/15083]

### Namespace collision: `--text-*`

[ASSUMED вЂ” needs smoke-test verification] Tailwind 4's `--text-*` namespace generates font-size utilities. The DS typography.css defines `--text-2xs`, `--text-xs`, etc. These names collide with Tailwind's default `text-xl`, `text-2xs` etc. scale. ### What to NOT put in `@theme inline` for Fibonacci spacing

[ASSUMED] The DS `--space-*` tokens use Fibonacci values (13px, 21px, 34px, 55px, 89px) that do not align with Tailwind's 4px spacing grid. ### Removing the `* { @apply transition-colors duration-150 }` anti-pattern

The current `app.css` has:
```css
* {
  @apply transition-colors duration-150;
}
```

This is a **known pitfall**: global `transition-colors` on every element causes theme-toggle transitions to fire on ALL elements simultaneously, creating visual churn. The round1 HTML scopes transitions only to shell elements (`html, body, .zt-topnav, .zt-bottomtab, .zt-theme-toggle`). Replace the global rule with targeted transitions on the shell layer only. ## Dark Theme in React Router 7 вЂ” SSR-Safe No-Flash

### The hydration mismatch problem

React SSR renders `<html lang="en">` with no `data-theme` attribute. The inline script sets `data-theme="dark"` on the `<html>` element before paint. When React hydrates, it finds `data-theme` on the real DOM but the server-rendered HTML does not have it в†’ hydration mismatch warning.

### Solution

[VERIFIED: medium.com/@almina.brulic/add-a-theme-toggleвЂ¦ + react.dev/reference/react-dom/client/hydrateRoot]

```tsx

          }}
        />
        <Meta />
        <Links />
      </head>
      <body className="bg-paper text-ink min-h-screen">

        <ScrollRestoration />

      </body>

```

Key points:
- `suppressHydrationWarning` on `<html>` silences the `data-theme` mismatch (one level deep вЂ” sufficient)
- The `<script>` tag is placed **before** `<Meta />` and `<Links />` so it fires before any CSS loads
- The script is minified inline (no `src` attribute) вЂ” it is a blocking script and must be tiny
- `dangerouslySetInnerHTML` is the correct React pattern for injecting a raw inline `<script>` without escaping
- On RR7, the Layout export is the root document вЂ” this is the only place to inject before-paint scripts [VERIFIED: reactrouter.com/api/framework-conventions/root.tsx]

### ThemeToggle component pattern

```tsx
// app/components/ui/ThemeToggle.tsx
import { Moon, Sun } from "lucide-react";
import { useState, useEffect } from "react";

export function ThemeToggle() {
  const [theme, setTheme] = useState<"light" | "dark">(() => {
    if (typeof document === "undefined") return "light";
    return (document.documentElement.getAttribute("data-theme") as "light" | "dark") || "light";
  });

  const toggle = () => {
    const next = theme === "dark" ? "light" : "dark";
    document.documentElement.setAttribute("data-theme", next);
    try { localStorage.setItem("zt-theme", next); } catch {}
    setTheme(next);
  };

  return (
    <button
      type="button"
      onClick={toggle}
      aria-label={theme === "dark" ? "Switch to light theme" : "Switch to dark theme"}
      title={theme === "dark" ? "Light theme" : "Dark theme"}
      className="inline-flex items-center justify-center w-10 h-10 rounded-md border border-border bg-surface text-text-muted transition-colors hover:text-ink hover:border-border-strong active:scale-92"
    >
      {theme === "dark" ? <Sun size={19} strokeWidth={1.9} /> : <Moon size={19} strokeWidth={1.9} />}
    </button>
  );
}
```

Note: Initial state reads from `document.documentElement` (set by the inline script), so on client there is no flash even on re-render. SSR always returns `"light"` fallback вЂ” the `suppressHydrationWarning` on `<html>` covers this.

## Porting Inline-Style `.jsx` в†’ Typed Tailwind TSX

### General pattern

1. Copy the component's shape from its `.d.ts` type signature (already typed for DS primitives; new components modeled from `lib.jsx` implementations)
2. Replace `style={{ ... }}` with Tailwind utility classes where brand tokens have `@theme`-generated utilities
3. Use `style={{ ... }}` for values that have NO `@theme` utility (e.g. `pathLength="1"` on SVG, computed pixel values for MetricRing, precise inline `strokeDashoffset`)
4. Replace `React.useState` / `React.useEffect` with named imports
5. Replace CDN `window.*` references and hash router `Link` / `navigate` with React Router 7 `Link` / `useNavigate`
6. Add TypeScript props interface (extend from `.d.ts` if available, or write new)
7. Add `aria-*` attributes per accessibility requirements (ThemeToggle, IconButton, BottomTab items)

### SVG components: keep CSS var strings, not Tailwind classes

SVG attributes (`stroke`, `fill`) accept string values, not CSS classes. Pass CSS variable strings directly:

```tsx
// MetricRing.tsx вЂ” correct pattern

<circle stroke={toneColor} ... />

This allows dark-mode remap to work automatically вЂ” `var(--vital)` stays teal in light, flips based on dark block. [VERIFIED: from MetricRing.jsx implementation]

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Dark-mode CSS variable cascade | Per-element `dark:` class on every element | `@theme inline` + `html[data-theme="dark"]` CSS block | Semantic token remap propagates to all utilities automatically |
| No-flash theme init | setTimeout-based JS toggle | Blocking inline `<script>` in `<head>` before `<Links />` | Only a before-paint script prevents FOUC; any async approach flickers |
| Icon rendering | Custom SVG sprite sheet | `lucide-react` named imports | Tree-shaken, typed, consistent stroke/size API |
| Chart theming | Recharts `customized` prop + inline style objects | `stroke="var(--ink)"` CSS var strings | Native CSS cascade, automatic dark mode, no component reimplementation |
| Mobile bottom nav | Custom CSS media breakpoints from scratch | The `zt-bottomtab` / `zt-topnav` CSS patterns from round1 HTML | Proven pattern with `env(safe-area-inset-bottom)`, 48px touch targets, correct z-index stacking |
| MetricRing animation | `requestAnimationFrame` loop | CSS `transition: stroke-dashoffset var(--dur-ring) var(--ease-out)` on the SVG circle | CSS transition is GPU-accelerated; the `.jsx` source already uses this pattern |
| Responsive grids | JavaScript breakpoint detection | Tailwind responsive utilities (`sm:`, `md:`) + the `zt-grid-*` pattern from round1 | Pure CSS; server renders correct markup |

**Key insight:** Every token-driven visual in this phase is a CSS cascade problem, not a JS state problem. The dark mode remap, grid reflow, nav visibility, and font switches all belong in CSS. Components own structure and interaction state; CSS owns appearance.

## Common Pitfalls

### Pitfall 1: `@theme` (not inline) with dark mode

**What goes wrong:** Declaring `--color-paper: #f5f2f0` in `@theme` (no `inline`) generates `bg-paper { background-color: #f5f2f0; }` with a static hex. The `html[data-theme="dark"]` block remaps `--paper: #15110f` but `bg-paper` never picks it up. Dark mode appears broken on all surfaces.
**Why it happens:** Without `inline`, Tailwind resolves variable references at compile time and inlines the final value.
**How to avoid:** Always use `@theme inline { --color-paper: var(--paper); }` for any token that participates in dark mode.
**Warning signs:** Dark mode toggle changes visible but surface/text colors don't change.

### Pitfall 2: Recursive `--font-*` naming collision

**What goes wrong:** `@theme inline { --font-display: var(--font-display); }` references itself. Depending on CSS parser evaluation order, this could create an infinite variable reference loop.
**Why it happens:** The DS token file and the `@theme` bridge use the same variable name.
**How to avoid:** Rename the DS token in `@theme` to alias from a separate var. Option A: In `app.css`, re-declare the DS token with a bridge alias BEFORE the `@theme` block: `:root { --ds-font-display: var(--font-display); }` then `@theme inline { --font-display: var(--ds-font-display); }`. Option B: Directly inline the font string value: `@theme inline { --font-display: 'Space Grotesk', ui-sans-serif, system-ui, sans-serif; }`. Option B is simpler and avoids the cascade entirely вЂ” use it. [ASSUMED вЂ” test both; Option B is the safer path]

### Pitfall 3: `--text-*` namespace clash with Tailwind type scale

**What goes wrong:** Tailwind 4's `--text-*` namespace also covers font-size utilities. Bridging `--text-xl: 1.75rem` into `@theme inline` overrides Tailwind's default `text-xl` (1.25rem). Existing routes that use `text-xl` for headings will change size unexpectedly.
**Why it happens:** DS type scale and Tailwind default type scale share the same namespace.
**How to avoid:** During Wave 3 route retrofit, audit all existing `text-*` usages before Wave 1 bridge lands. Alternatively, bridge DS type tokens under a prefixed name (`--text-ds-xl`) and use them directly in component CSS. Since all routes are being retrofit anyway, overriding the defaults is acceptable вЂ” just do it in Wave 1 and ensure Wave 3 route work uses DS values consistently.
**Warning signs:** Build passes but visually wrong font sizes on retrofit screens.

### Pitfall 4: `* { @apply transition-colors }` causing theme toggle visual churn

**What goes wrong:** The existing `app.css` applies `transition-colors duration-150` globally. When the theme toggle fires, every element in the DOM simultaneously transitions, causing a cascading visual flash вЂ” especially with 200+ DOM elements on a metrics detail page.
**Why it happens:** Global CSS transitions fire on every element that has a changing color property.
**How to avoid:** Remove the global `* { @apply transition-colors duration-150; }` rule. Apply transitions only to shell elements (`.zt-topnav, body, html, .zt-bottomtab, .zt-theme-toggle`) as done in the round1 HTML.
**Warning signs:** Theme toggle looks "slow" or has visible color cascade spreading across the page.

### Pitfall 5: `react-router typegen` not run before `tsc`

**What goes wrong:** Adding new route components that import from `./+types/root` or `./+types/{route}` fail TypeScript compilation with "module not found" errors on first run.
**Why it happens:** RR7 generates route types into `.react-router/types/` at build time; the directory may not exist in a fresh clone.
**How to avoid:** Always run `npm run typecheck` (which runs `react-router typegen && tsc`) вЂ” never run `tsc` alone. In CI, ensure typegen runs first. [VERIFIED: reactrouter.com/api/other-api/dev]
**Warning signs:** `tsc` errors on `./+types/root` import at the top of new route files.

### Pitfall 6: SSR crash on `lucide-react` icon in server context

**What goes wrong:** Some older icon libraries reference `document` or `window` during import вЂ” SSR crashes with "document is not defined".
**Why it happens:** Icons that use DOM APIs at module level fail in Node.js.
**How to avoid:** `lucide-react` 1.x renders pure SVG elements, no DOM access at module level. [VERIFIED: lucide.dev] Safe to use in SSR. Still, add an early smoke test: import one icon in a route loader stub and confirm `npm run typecheck` and `npm run build` pass.
**Warning signs:** Build error like `ReferenceError: window is not defined` during SSR bundle generation.

### Pitfall 7: Recharts `ResponsiveContainer` in SSR

**What goes wrong:** `ResponsiveContainer` with `width="100%"` outputs 0Г—0 in SSR because there is no DOM to measure. On hydration, it resizes вЂ” causing a brief layout shift.
**Why it happens:** Recharts uses ResizeObserver internally.
**How to avoid:** For metric detail charts with fixed height, use `<ResponsiveContainer width="100%" height={height}>` where height is always a concrete number (not a percentage). The existing `TrendChart.tsx` already does this correctly. [VERIFIED: from existing code review]
**Warning signs:** Chart area is collapsed to 0px height on initial server-rendered page, then "pops" in.

### Pitfall 8: `color-mix()` in TopNav backdrop for older browsers

**What goes wrong:** The TopNav uses `background: color-mix(in srgb, var(--paper) 86%, transparent)` for the glassmorphism effect. Safari < 16.2 does not support `color-mix()`.
**Why it happens:** `color-mix()` is a CSS function with partial browser support.
**How to avoid:** Add a solid fallback: `background: var(--paper); background: color-mix(...)`. The `backdrop-filter` blur already has a `-webkit-backdrop-filter` prefix in the round1 CSS вЂ” apply both. For the target audience (practitioner instrument, not consumer), modern browser support is acceptable. [ASSUMED вЂ” check actual browser targets for the project]

## Open Questions (RESOLVED)

> All three resolved during planning вЂ” resolutions embedded in plan task actions
> (Plan 04.1-02 Task 1 for A1/A7, Plan 04.1-03 Task 3 for the chart animation),
> each with a documented fallback. None is an execution blocker.

1. **`@theme inline` self-referential `--font-display` variable** вЂ” **RESOLVED (Plan 04.1-02 Task 1).**
   - What we know: `@theme inline { --font-x: var(--font-x) }` is technically a CSS variable that references itself. The Tailwind documentation does not explicitly cover this case.
   - What's unclear: Does Tailwind 4.1 resolve this to the already-declared `:root` value, or does it create an empty variable?
   - **Resolution:** Inline the font-stack strings directly in `@theme inline` (e.g. `--font-display: 'Space Grotesk', ui-sans-serif, system-ui, sans-serif;`) rather than self-referencing `var(--font-display)`. Wave 1 smoke-test (build + inspect generated CSS) confirms `font-display` utility resolves; direct inlining is the safe default.

2. **Token import path resolution from `app.css`** вЂ” **RESOLVED (Plan 04.1-02 Task 1).**
   - What we know: `app.css` is at `remix-app/app/app.css`. Token files are at `docs/design-system/tokens/`. Relative path `../../docs/design-system/tokens/colors.css` navigates above `remix-app/`.
   - What's unclear: Whether `@tailwindcss/vite` follows CSS `@import` path resolution relative to the CSS file or the Vite root.
   - **Resolution:** Wave 1 tests the `@import`; if it fails (path escapes the Vite root), the fallback is to inline the token file contents directly into `app.css` (they are short: 107 / 49 / 73 lines). Either way the `:root` vars land before the `@theme inline` bridge.

3. **Exact Recharts animation API for line sweep** вЂ” **RESOLVED (Plan 04.1-03 Task 3).**
   - What we know: Recharts `<Line>` supports `animationDuration` (ms) and `animationEasing`. The round1 TrendChart uses a pure SVG `pathLength=1` / `stroke-dashoffset` CSS keyframe.
   - What's unclear: Whether Recharts' built-in animation matches the `zt-draw` visual quality.
   - **Resolution:** Accept Recharts' built-in animation; `zt-draw` may additionally wrap page-load charts. Exact keyframe fidelity is explicitly not a hard requirement for Phase 04.1 (Claude's Discretion).
