<objective>
Retrofit the data-access seam so every tenant/subject-scoped DB interaction flows through `withTenantDb(ctx, fn)` (TEN-03): convert the read functions in `data.server.ts`, the consent gate in `consent.server.ts`, and the audit writer in `audit.server.ts` to wrap their bodies in `withTenantDb`, change their signatures to accept a `TenantCtx`, and update all ~15 loader/action call sites accordingly. Background/no-subject contexts (the `extractionWorker`, auth events) use a getDb() admin path that intentionally bypasses RLS.

Purpose: This is the live wiring that makes RLS actually govern the running app — without it, RLS is enabled but never engaged because queries run as neondb_owner via getDb(). It honors the "Phase 7 withTenantDb retrofit boundary" comments already coded into the three service files.
Output: data/consent/audit service functions wrapped in withTenantDb; all call sites pass TenantCtx; owner data renders identically (parity preserved); RLS is now the enforced backstop.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Loader/action → service fn | The TenantCtx passed in is the claimed identity; withTenantDb sets it as the RLS context |
| Background worker (no session) → DB write | extractionWorker has no request session; its writes use the admin path and must not be reachable with attacker-controlled tenant data |
| Approve write (metrics INSERT) → RLS | The review approve path writes PHI; WITH CHECK must validate the tenant_id at insert |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-10 | Spoofing | TenantCtx forged at call site | mitigate | ctx is derived only from the requireUser session (user.id/user.tenantId) + getOwnerSubject — never from request body; app-layer authz (requireUser/assertSubjectAccess) gates before withTenantDb |
| T-07-11 | Information Disclosure | Read bypasses RLS via leftover getDb() | mitigate | data.server.ts contains exactly one getDb() (getOwnerSubject, tenant-only subjects lookup); all entity reads go through withTenantDb — verified by grep gate in Task 1 |
| T-07-12 | Elevation of Privilege | Approve-write bypasses RLS | mitigate | review.tsx approve transaction moved into withTenantDb(ctx); WITH CHECK rejects a metrics INSERT with a mismatched tenant_id; assertSubjectAccess still precedes the write |
| T-07-13 | Repudiation | Audit write silently dropped under RLS | mitigate | audit_log INSERT policy (WITH CHECK on app.tenant_id) is satisfied by the withTenantDb path; background-job writes use the admin path that is not RLS-blocked |
| T-07-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this plan; supply-chain surface unchanged |
</threat_model>
