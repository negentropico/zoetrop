<objective>
Close AUTH-03 Blocker 1 (CR-01 dead code). `listAssignedSubjectIds` (remix-app/app/lib/assignments.server.ts) is correctly implemented but has zero call sites: every `assertSubjectAccess` call in every practitioner-admitting route passes only 3 arguments, so the optional 4th `assignedSubjectIds` parameter is always `undefined` and Gate 3 (the per-assignment practitioner check) is permanently skipped at runtime. An unassigned practitioner can upload PDFs, approve extractions, generate reports, and read reports for any subject in the tenant.

This plan wires `listAssignedSubjectIds(ctx, user.id)` into all five practitioner-admitting routes, passes the result as the 4th argument to `assertSubjectAccess`, and fixes the two call sites that do not currently pass a `subject.id` (review.tsx action, reports/detail.tsx loader) so Gate 3 has a subject id to check.

Purpose: Make the per-assignment practitioner gate (AUTH-03, D-07) actually enforce at runtime. The assignment table, RLS policy, assignment-management UI, and the unit-tested assertSubjectAccess Gate 3 logic already exist — only the route→lookup wiring is absent.
Output: Five route files wired to the assignment lookup + a unit test file proving the deny/pass matrix.
</objective>

<usage_note>
The current 3-arg calls all pass `undefined` for assignedSubjectIds, which makes Gate 3 a no-op. The fix in every route is identical in shape:

  const assignedIds = user.role === "practitioner"
    ? await listAssignedSubjectIds(ctx, user.id)
    : undefined;
  assertSubjectAccess(user, subject, user.tenantId!, assignedIds);

- For owners, assignedIds stays undefined → owners keep tenant-wide access (D-07, no regression).
- For practitioners, assignedIds is their active assigned subject set → Gate 3 denies unless subject.id is in it.
- `ctx` (TenantCtx) is already constructed in every route; the lookup reuses it. Where the access check currently runs BEFORE ctx is built, move the ctx construction up so the lookup can run.
- The `subject` object passed to assertSubjectAccess MUST carry an `id`. Two call sites currently pass an object with `tenantId` only — those must be given a subject id (extraction.subjectId / report.subjectId).
</usage_note>
