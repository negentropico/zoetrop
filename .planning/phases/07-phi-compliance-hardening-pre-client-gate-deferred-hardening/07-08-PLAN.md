<objective>
Close Blocker CR-02: the `practitioner_subject_assignments` unique index is a FULL unique index, not
a partial index on `revoked_at IS NULL`. After any revocation the soft-deleted row permanently
occupies the unique key, so re-assigning the same (tenant, practitioner, subject) triple hits the
unique constraint. The `assignSubject` catch block matches the violation and returns
`{ assigned: true, alreadyExists: true }` — a FALSE success — while `listAssignedSubjectIds` (filters
`revoked_at IS NULL`) returns nothing. The owner-facing /settings/assignments UI is therefore
one-shot per (practitioner, subject) pair: a single revoke permanently breaks re-granting access.

Fix: make the index partial (`WHERE revoked_at IS NULL`) so only ACTIVE rows participate in
uniqueness. A revoked row no longer blocks a fresh active assignment. This requires:
(a) schema.ts — add a `.where(sql\`revoked_at IS NULL\`)` predicate to the uniqueIndex,
(b) a new migration that drops the full index and creates the partial index,
(c) [BLOCKING] applying that migration to the live Neon database — the build/typecheck pass
    without the live apply (types come from config, not the live DB), so skipping it produces a
    false-positive verification state,
(d) assignments.server.ts — replace fragile substring error matching with a structured Postgres
    error-code check (23505 = unique_violation),
(e) a regression test proving the assign->unassign->assign round-trip works end-to-end.

Purpose: the assignment management lifecycle works repeatedly (assign / revoke / re-assign), which
is required for AUTH-03 SC-4 (a functional owner-facing assign/unassign UI).
Output: partial index live on Neon, structured error matching, round-trip regression test.
</objective>

## Trust Boundaries

| owner assign/unassign action -> practitioner_subject_assignments | The owner manages which subjects a practitioner can reach; if the lifecycle silently breaks, an intended grant produces NO access (fail-broken authz) |
| Postgres unique constraint -> assignSubject result | The DB constraint outcome is interpreted by the catch block; a misinterpretation converts a real failure into a false success |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-08-01 | Denial of Service (authz lockout) | idx_psa_active_unique full index | mitigate | Replace full index with PARTIAL index `WHERE revoked_at IS NULL` (Task 1) and apply to live Neon (Task 2). A revoked row no longer occupies the unique key, so an owner can re-grant access after a revoke. Without this, an intended grant is permanently denied (fail-broken). |
| T-07-08-02 | Spoofing / Tampering (false success) | assignSubject catch block | mitigate | Structured 23505 matching (Task 3) so a real constraint failure is not masked as `{ assigned: true }`. With the partial index, 23505 only fires for a genuinely active duplicate; the UI's "Already assigned" message becomes truthful. |
| T-07-08-03 | Tampering | live migration apply | mitigate | [BLOCKING] human-verified live apply (Task 2) with `pg_indexes.indexdef` proof so the schema source-of-truth and the live DB agree; prevents a false-positive verification where types pass but the DB index is still full. |
