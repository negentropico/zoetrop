## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| App role → table data | The app_user role's RLS policies are the DB-layer isolation backstop; a missing/weak policy leaks PHI cross-tenant |
| Service/admin path (neondb_owner) → RLS | Better-Auth + corpus + migrations run as the owner role which BYPASSES RLS — must be confined to non-PHI / pre-context tables |
| Migration apply → live PHI tables | A bad atomic migration could break reads for the live owner or silently leave a table RLS-free |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-04 | Information Disclosure | Cross-tenant read via RLS gap | mitigate | ENABLE+FORCE RLS + USING predicate on all 16 PHI tables keyed on app.tenant_id/app.subject_id; cross-tenant zero-read proven by rls-isolation.test.ts (Task 4) |
| T-07-05 | Tampering | Forged-tenant INSERT | mitigate | WITH CHECK clause on every policy mirrors USING; mismatched-tenant INSERT rejected (proven by the WITH CHECK test) |
| T-07-06 | Elevation of Privilege | RLS bypass via owner/service role | mitigate | app_user is NOSUPERUSER NOBYPASSRLS; FORCE RLS applies even to table owners; PHI queries never run as neondb_owner (withTenantDb does SET LOCAL ROLE app_user) |
| T-07-07 | Tampering / Repudiation | Audit-log tampering | mitigate | audit_log gets INSERT+SELECT policies ONLY (no UPDATE/DELETE policy) + REVOKE UPDATE,DELETE — rows are immutable for app_user (AUTH-04/D-08) |
