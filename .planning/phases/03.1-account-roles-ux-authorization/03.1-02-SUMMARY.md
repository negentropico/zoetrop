## What Was Built

### `remix-app/app/lib/invites.server.ts`
Exports the complete invite token lifecycle:
- `hashToken(raw)`: SHA-256 hex, 64 chars, deterministic
- `newRawToken()`: 32-byte base64url (256-bit, URL-safe, opaque)
- `generateInvite(opts)`: policy-gated via `can()`, tenant-scoped, inserts only the hash (never raw), 7-day expiry, returns raw token once
- `listInvites(tenantId)`: tenant-scoped SELECT ordered newest-first
- `revokeInvite({ id, tenantId })`: tenant+id scoped UPDATE (IDOR guard), idempotent
- `consumeInviteByToken(raw)`: hashes raw → SELECT (not-consumed, not-revoked, not-expired) → guarded UPDATE WHERE consumedAt IS NULL → returns `{ role, tenantId }` or null. Catch-all returns null (fail-closed, T-031-INV-6).

**Policy enforcement (D-10):** delegated to `can(inviter, "invite:practitioner"|"invite:client")` from Plan 01's authz.server.ts.

### `remix-app/app/lib/auth.server.ts` (modified)
Rewritten `beforeSignUp` hook with:
1. Break-glass: `OWNER_INVITE_TOKEN` timing-safe-equal check first (bootstrap-only, role defaults to "client")
2. Per-invite: `consumeInviteByToken(token)` DB lookup; null → FORBIDDEN (fail-closed)
3. On valid invite: stashes `{ role, tenantId }` in module-level `pendingInviteRoles` Map keyed by normalized email

New `databaseHooks.user.create.before`: reads from `pendingInviteRoles`, overrides `role` + `tenantId` before the user row is written. Entry is deleted immediately after use.

`tenantId` added as `input: false` additional field so it can be set server-side.

### `remix-app/tests/auth/invites-server.test.ts`
13 test cases covering:
- hashToken: 64-char hex, deterministic, collision-resistance
- SHA-256 of "test" = known value (ground truth)
- newRawToken: high-entropy, URL-safe, different across calls, ≥40 chars
- generateInvite: policy rejection (practitioner→practitioner: 403, client→any: 403, null tenantId: throws)
- consumeInviteByToken: unknown token → null (fail-closed), all exports exist (structural)
- DB-dependent tests skipped via `it.skipIf(!hasDb)` when DATABASE_URL* absent

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing Critical Functionality] tenantId not declared as additionalField in auth.server.ts**
- **Found during:** Task 2 — wiring databaseHooks injection
- **Issue:** `tenantId` exists in the DB (Plan 01 migration added `user.tenant_id`) but was not declared in Better-Auth's `user.additionalFields`. Without this declaration, Better-Auth's `parseUserOutput()` strips it from session responses.
- **Fix:** Added `tenantId: { type: "string", required: false, defaultValue: null, input: false }` to `user.additionalFields` in auth.server.ts.
- **Files modified:** remix-app/app/lib/auth.server.ts
- **Commit:** 787f934

**2. [Rule 1 - Bug] Role injection requires databaseHooks, not ctx.body modification**
- **Found during:** Task 2 — reading better-auth node_modules (plan directed this)
- **Issue:** The plan mentioned "inject extra user fields via ctx.context" but `parseUserInput()` in better-auth explicitly blocks `input: false` fields from the request body (throws FIELD_NOT_ALLOWED). Setting role in ctx.body would fail at runtime.
- **Fix:** Used `databaseHooks.user.create.before` (same pattern as the better-auth admin plugin) with a module-level `pendingInviteRoles` Map to bridge the before-request hook and the DB-level hook. This is the documented server-side injection mechanism.
- **Files modified:** remix-app/app/lib/auth.server.ts
- **Commit:** 787f934
