## Summary

This phase delivers an invite-only auth/tenancy surface: hashed single-use invite tokens, fail-closed role/capability checks, a Better-Auth `beforeSignUp` invite gate, and the account-settings/invite UI. The core security spine is mostly sound — token hash-at-rest, the race-safe guarded UPDATE on consume, fail-closed `requireRole`/`can`, tenant-scoped revoke, and `input:false` on role/tenantId all hold up under tracing.

Two findings are BLOCKER-class. The most serious is a **race window in the `beforeSignUp` → `databaseHooks.user.create.before` handoff via a module-level `Map` keyed by email** (CR-01): the invite is consumed (marked single-use) in `beforeSignUp`, but the resolved role/tenantId is delivered to the user-create write through a process-global `Map`. On serverless this is brittle, and the consume-then-stash sequence is not atomic with the user write — failure modes silently drop the role/tenant (privilege downgrade to a tenant-less `client`) **after** the single-use token is already burned, leaving no recovery path. The second BLOCKER is that the **break-glass path returns before stashing any tenant**, so a break-glass signup creates a `client` user with `tenantId: null` (CR-02) — and unlike the seed script, nothing repairs it.

The remaining warnings concern an open hole where `consumedBy` is never written (audit/forensic gap), no `name`-field validation on profile update, a tenant-less actor passing `requireRole` then silently revoking nothing, and several robustness gaps. Info items are naming/dead-comment cleanups.

## Critical Issues

### CR-01: Invite consumed before role/tenant delivery is guaranteed — non-atomic handoff via process-global Map drops privileges and burns the token

**File:** `remix-app/app/lib/auth.server.ts:32`, `87-111`, `129-186`
**Issue:**
The `beforeSignUp` middleware calls `consumeInviteByToken(token)` (line 161), which **immediately marks the invite consumed** (single-use UPDATE). It then stashes `{ role, tenantId }` into a module-level `pendingInviteRoles` Map keyed by `email.toLowerCase()` (line 179). A *separate* hook, `databaseHooks.user.create.before` (line 90), later reads that Map and injects the role/tenant onto the user row.

Two distinct failures fall out of this design:

1. **Non-atomic, lossy handoff.** If anything between consume and the user-row write fails or doesn't line up — the user-create hook reading a stale/empty Map, an email-normalization mismatch, a duplicate-email rejection *after* consume, or a second serverless instance handling the create — the invite is **already burned** but the role/tenant is **never applied**. The result is a `client` user with `tenantId: null` and an invite that can never be reused. There is no compensating action (no un-consume, no transaction). The consume must not happen until the user row is committed with the injected role/tenant.

2. **Process-global mutable state on serverless.** The Map lives in module scope (`const pendingInviteRoles = new Map(...)`, line 32). The comment claims "Node.js is single-threaded; Map access here is safe," but Vercel runs multiple isolated lambda instances. `beforeSignUp` and `user.create.before` *happen* to run in the same request here, but the pattern is fragile: any future code path that crosses an instance boundary, or concurrent signups colliding on the same normalized email key, silently corrupts role assignment. A leaked/abandoned entry (signup that errors after stash but the Map delete in `create.before` never runs) is a small memory leak and a stale-role landmine for the next same-email signup.

**Fix:** Do not consume the token in `beforeSignUp`. Resolve+validate the invite there (a non-mutating lookup), and perform the single-use consume **inside the same DB transaction** that writes the user row (or in `databaseHooks.user.create.before`/`after`, immediately adjacent to the write), so consume and role-injection commit or roll back together. Pass the resolved invite via request-scoped context (`ctx` state) rather than a module-global Map. At minimum, move `consumeInviteByToken` into `user.create.before` so the burn and the role-write are in the same step:

```ts
// beforeSignUp: validate only (no consume), stash raw token in ctx-scoped state
// user.create.before: consume + inject atomically
async before(user) {
  const email = typeof user.email === "string" ? user.email.toLowerCase() : null;
  const rawToken = /* request-scoped, not module Map */;
  if (!email || !rawToken) return undefined;
  const invite = await consumeInviteByToken(rawToken); // burn here, adjacent to write
  if (!invite) throw new APIError("FORBIDDEN", { message: "signup_disabled" });
  return { data: { ...user, role: invite.role, tenantId: invite.tenantId } };
}
```

### CR-02: Break-glass signup creates a tenant-less `client` with no repair path — silently unusable account from a burned bootstrap token

**File:** `remix-app/app/lib/auth.server.ts:142-150`
**Issue:**
The break-glass branch returns early (line 149) without stashing anything in `pendingInviteRoles`. The `databaseHooks.user.create.before` hook then finds no pending entry and applies the defaults: `role: "client"`, `tenantId: null`. The seed script (`scripts/seed-owner.ts:111-114`) works around this by issuing a direct Drizzle `UPDATE` to set `role: "owner"` + `tenantId` *after* signup — but that repair only runs inside the seed script. **Any other use of `OWNER_INVITE_TOKEN` to sign up** (the comment at lines 138-140 explicitly says the token is "Retained for bootstrap-only: an already-seeded owner can use this token to permit a single additional signup") produces a `client` account with `tenantId: null`. Such an account passes the `_app/layout.tsx` session gate but has no tenant, so every tenant-scoped feature (invites loader returns `[]`, `assertSubjectAccess` denies) silently fails for that user with no error surfaced and no path to assign a tenant through the UI.

Worse, the break-glass token is a single static shared secret with **no single-use enforcement and no expiry** — it can mint unlimited tenant-less `client` accounts as long as the env var is set, each one a dead-end account. This is a footgun masquerading as a feature.

**Fix:** Either (a) remove the break-glass path entirely now that per-invite seeding works (the seed script could mint and consume a real invite row instead), or (b) make break-glass require an explicit `tenantId` (e.g. a second env var `OWNER_TENANT_ID`) and stash `{ role: "owner", tenantId }` so the account is actually usable, and gate the whole branch behind a `NODE_ENV !== "production"` or first-user-only check so it cannot be left live in production. Do not ship a path that creates accounts the application cannot operate on.

## Warnings

### WR-01: `consumedBy` is never written — broken single-use audit trail and dangling nullable column

**File:** `remix-app/app/lib/invites.server.ts:201-217`, `remix-app/db/schema.ts:89`
**Issue:**
The schema defines `consumedBy: text('consumed_by').references(() => user.id)` (and migration 0005 adds the FK), documented as part of the invite audit trail. But `consumeInviteByToken` only sets `consumedAt` (line 203) — it never records *who* consumed the invite. The function signature can't even know the consuming user yet (it runs in `beforeSignUp` before the user row exists). The result is a permanently-NULL column that defeats the forensic purpose it was added for: after consumption you cannot tell which user redeemed which invite, which matters for a multi-tenant invite system (detecting invite-sharing, attributing accounts to inviters).

**Fix:** Populate `consumedBy` in the same step that creates the user (tie it to CR-01's fix — once consume happens adjacent to the user-create write, the new user id is available). Set `consumedBy: newUser.id` in the consume UPDATE, or backfill it in `databaseHooks.user.create.after`. Until then the column is dead weight and the audit contract in the schema comment is false.

### WR-02: Tenant-less owner/practitioner passes `requireRole` then silently revokes nothing (degraded auth state, no error)

**File:** `remix-app/app/routes/_app/settings/invites.ts:24-32`
**Issue:**
The revoke action calls `revokeInvite({ id: params.inviteId!, tenantId: user.tenantId ?? "" })`. If `user.tenantId` is null/undefined (a real state — see CR-02, and `auth-schema.ts:26` makes `tenantId` nullable), the empty-string tenant matches zero rows, so the UPDATE is a silent no-op that still returns `{ success: true }`. The user sees the invite flip to "revoked" in the UI (`InviteTableRowAction` optimistically renders revoked on `fetcher.data.success`, `index.tsx:884-887`) while the DB row remains active. The invite is still redeemable. This is a correctness/security gap: the UI lies about revocation state.

This is partly masked because the settings loader only lists invites when `canInviteClient` (which requires `!!user.tenantId`, `index.tsx:36`), so a tenant-less user normally sees no invites to revoke — but the resource route at `/settings/invites/:id/revoke` is directly reachable by any authenticated owner/practitioner regardless of the loader gate.

**Fix:** Reject explicitly when the actor has no tenant, before calling `revokeInvite`:

```ts
if (!user.tenantId) {
  throw new Response("No tenant assignment.", { status: 403 });
}
await revokeInvite({ id: params.inviteId!, tenantId: user.tenantId });
```

Also have `revokeInvite` (or the action) verify a row was actually updated (`.returning()` and check length) so a no-op revoke surfaces an error instead of a false success.

### WR-03: `params.inviteId!` non-null assertion can yield `id: undefined` filter — defensive validation missing

**File:** `remix-app/app/routes/_app/settings/invites.ts:25-26`
**Issue:**
`id: params.inviteId!` uses a non-null assertion. While the route pattern `settings/invites/:inviteId/revoke` normally guarantees the param, a malformed match or future route refactor could pass `undefined`, producing `eq(invites.id, undefined)` in the Drizzle filter. Depending on the driver this either throws (caught by the bare `catch`, returning a generic "Try again" that masks the real cause) or matches nothing. Combined with WR-02's false-success, a bad id silently appears to succeed in the UI.

**Fix:** Validate the param explicitly: `if (!params.inviteId) throw new Response("Not found", { status: 404 });` before constructing the revoke call.

### WR-04: `update-profile` action accepts unvalidated `name` (empty/whitespace/non-string) and casts blindly

**File:** `remix-app/app/routes/_app/settings/index.tsx:57-72`
**Issue:**
`update-profile` does `const name = formData.get("name") as string;` and passes it straight to `auth.api.updateUser({ body: { name } })`. A crafted POST with no `name` field yields `null` cast to `string` (TS lies; runtime is `null`), and an empty/whitespace name passes through. The `user` table requires `name` NOT NULL (`auth-schema.ts:12`), so a `null` name could throw (caught generically) or, worse, an empty string blanks the display name with no validation. The login signup path correctly validates name (`login.tsx:42-44`) — this action does not, an inconsistency.

**Fix:** Validate before calling updateUser:

```ts
const name = formData.get("name");
if (typeof name !== "string" || !name.trim()) {
  return { intent: "update-profile", success: false, error: "Please enter your name." };
}
await auth.api.updateUser({ body: { name: name.trim() }, headers: request.headers });
```

### WR-05: `generate-invite` form trusts client-supplied `role` for practitioner invites without re-checking `invite:practitioner`

**File:** `remix-app/app/routes/_app/settings/index.tsx:123-142`
**Issue:**
The action gates with `requireRole(user, ["owner", "practitioner"])` (line 125) then reads `role` from the form (line 127). A **practitioner** passes `requireRole`, and can craft a POST with `role=practitioner` (the client UI hides the practitioner button for them, `index.tsx:639`, but that is cosmetic). The action validates only that `role` is `"practitioner" | "client"` (line 128) — it does **not** re-check that this specific actor may invite *that specific role*. The actual defense is one layer down: `generateInvite` calls `can(inviter, "invite:${role}")` (`invites.server.ts:81`), which correctly denies practitioner→practitioner. So the system is safe **today**, but the route handler is relying entirely on an implicit downstream check while appearing to do its own authorization with `requireRole`. This is defense-in-depth debt: the action's gate is misleadingly coarse (role-presence) versus the capability it actually needs (`invite:practitioner` vs `invite:client`).

**Fix:** Re-assert the capability for the requested role in the action, so authorization intent is explicit at the boundary and not solely dependent on `generateInvite` internals:

```ts
if (!can(user, `invite:${role}`)) {
  throw new Response("You can't invite that role.", { status: 403 });
}
```

### WR-06: Settings `loader` exposes the full account-settings page to clients (no role gate); invite section hidden only client-side

**File:** `remix-app/app/routes/_app/settings/index.tsx:30-47`
**Issue:**
The settings `loader` only calls `requireUser` — there is no `requireRole`. The invite UI is conditionally rendered behind `canInviteClient` (a loader boolean, line 619), and the invite *action* is gated. That is correct for invites. But the prompt frames the settings surface as "owner/practitioner only; client → 403". As implemented, a `client` **can** load `/settings` (profile, password, preferences) — which may be the intended product behavior (clients manage their own profile), but it contradicts the stated contract. If clients are meant to be 403'd from the entire settings surface, the gate is missing. If clients are allowed (profile self-management), the contract/comment is wrong. Either way there is a mismatch between the documented authorization model and the code that should be reconciled before ship.

**Fix:** Decide and encode the contract. If settings is client-accessible, update the phase contract/comments to say so. If not, add `requireRole(user, ["owner", "practitioner"])` to the loader. Do not leave the authorization intent ambiguous on a security-sensitive surface.

### WR-07: `change-password` error mapping is dead/confused — all branches return the same string and swallow the real failure

**File:** `remix-app/app/routes/_app/settings/index.tsx:105-119`
**Issue:**
The catch builds `msg` from a ternary where **every branch returns "Current password is incorrect."** (lines 108-112). The `.includes("invalid")` / `.includes("password")` discrimination is pointless dead logic, and — more importantly — it **misreports any non-credential failure** (network error, Better-Auth internal error, rate-limit) as "Current password is incorrect," misleading the user and hiding real faults. A user with the *correct* current password but a transient backend error is told their password is wrong.

**Fix:** Differentiate genuine wrong-password from other failures, or at least don't assert a false cause:

```ts
} catch (err) {
  const wrongPw = err instanceof Error && /invalid|incorrect|password/i.test(err.message);
  return {
    intent: "change-password",
    success: false,
    error: wrongPw ? "Current password is incorrect." : "Couldn't change password. Try again.",
    errorField: wrongPw ? "currentPassword" : null,
  };
}
```
