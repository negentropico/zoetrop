# Phase 03.1: Account & Roles — UX + Authorization - Context

**Gathered:** 2026-06-09
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 3.1 builds the **account surface** and the **role-based authorization model** on top of the Phase 3 identity engine (Better-Auth email/password sign-in, the `owner`/`practitioner`/`client` role enum, the invite-only `beforeSignUp` gate, and the `tenants`/`users`/`subjects` spine with `tenantId`/`subjectId` on all 8 data tables). It was promoted from backlog 999.1 and **inserted before Phase 4** to close the day-to-day usability gap (there is currently no logout UI) and to make the platform multi-user-onboardable.

**In scope (all three captured items — owner chose full scope):**
1. **Account navigation** — an avatar/initials dropdown in the authenticated shell (`TopNav` top-right, mirrored into the mobile `BottomTab`) exposing Account/Settings, the theme toggle, and a Logout affordance that POSTs to the existing `/logout` action.
2. **Settings page** — a sectioned account hub. Functional now: Profile (name + email), Password change, Invite panel, Theme preference. Scaffolded as visible "Coming soon" placeholders (IA locked, wired up in later phases): Integrations, Data export, Privacy/Consent, Units/Display.
3. **Invite system** — a hand-rolled `invites` table replacing the single shared `OWNER_INVITE_TOKEN` check; single-use, role-scoped, expiring tokens; copy-link delivery; generate/list/revoke UI.
4. **Authorization model** — reusable server-side authz helpers enforcing a role→capability matrix in route loaders/actions, designed to be RLS-compatible so the Phase 7 `SET LOCAL`/RLS retrofit mirrors it.

**Out of scope (→ Phase 7, do NOT pull forward):** RLS enable + policies and the `SET LOCAL` `withTenantDb` enforcement wrapper, cross-tenant isolation tests (TEN-02/03), **practitioner per-assignment subject-scoping (AUTH-03)**, and the immutable auth/access **audit log (AUTH-04)**. Phase 3.1 enforces capabilities at the application layer only; DB-level isolation and the audit trail remain Phase 7.

**Explicitly parked (→ project-level decision, see Deferred):** migrating the platform to Supabase.

</domain>

<decisions>
## Implementation Decisions

### Scope shape
- **D-01:** **Full scope in 3.1** — account nav + settings, the invite system, AND role-based authorization enforcement all ship in this phase (owner overrode the lighter "UX-slice-only" recommendation). Accepted tradeoff: the authz enforcement partially front-runs the Phase 7 RLS work and is built before any non-owner user exists to exercise it. Mitigation: design authz to be **RLS-compatible** (app-layer checks Phase 7 can mirror with `SET LOCAL`/RLS claims, not contradict).

### Account navigation + shell
- **D-02:** **Avatar/initials dropdown menu**, top-right in `TopNav` where the `ThemeToggle` currently sits, opening a dropdown with **Account/Settings**, the **theme toggle**, and **Logout**. Logout is a `<Form method="post" action="/logout">` affordance (the action already exists and clears the cookie). Requires a small dropdown/menu component (none exists yet — build in `app/components/ui/`).
- **D-03:** The account entry is **mirrored into the mobile `BottomTab`** so logout/settings are reachable on mobile (≤760px, where the desktop nav cluster is hidden).

### Settings page
- **D-04:** Settings is a **sectioned hub**. **Functional now:** Profile (display name + email from the Better-Auth user record), Password change (Better-Auth `changePassword`, requires current password), Invite panel (generate/list/revoke), Theme preference (relocate/mirror the toggle).
- **D-05:** **Forward-looking sections are scaffolded as visible "Coming soon" placeholders** to lock the information architecture: Integrations, Data export, Privacy/Consent, Units/Display. Each is wired up when its underlying feature lands (Integrations/Consent → Phase 5; Export → post-Phase-4; Privacy → Phase 5/7; Units → when live data renders). Do **not** build their functionality now.

### Invite system
- **D-06:** **Hand-rolled `invites` table** — NOT the Better-Auth organization plugin (Phase 3 built the tenancy spine by hand; the org plugin would remap tenancy onto its org/member tables and contradict the existing model). Suggested columns: `token_hash`, `role` (`practitioner`|`client`), `tenant_id`, `created_by` (user id), `expires_at`, `consumed_at`. Store a **hash** of the token, never the raw token.
- **D-07:** The `beforeSignUp` hook is **rewritten** to look up the hashed token → verify not-consumed and not-expired → assign the encoded `role` + `tenant_id` on the new user → mark `consumed_at`. This **replaces** the single `OWNER_INVITE_TOKEN` constant-time check currently in `auth.server.ts`. (Planner: decide whether to keep `OWNER_INVITE_TOKEN` as a break-glass bootstrap fallback or remove it entirely — owner is already seeded.)
- **D-08:** **Tokens are single-use, role-scoped, and expiring.** Exact expiry window and token format are Claude's discretion (sensible default: opaque high-entropy token, ~7-day expiry).
- **D-09:** **Delivery = copy-link** (generate → show a copyable invite URL; owner/practitioner sends it manually). **No email infrastructure in 3.1.** Structure the invite-create path so an email-send (a provider or a future Supabase) can be added later without rework.
- **D-10:** **Invite policy = owner+practitioner tiered.** Owner can invite practitioners and clients; a practitioner can invite only clients into their own tenant; clients cannot invite.

### Authorization model
- **D-11:** **Reusable server helpers** in a new `app/lib/authz.server.ts` — `requireUser(request)`, `requireRole(session, roles)`, `assertSubjectAccess(session, subjectId)` — called at the top of route loaders/actions. Single source of truth, unit-testable, and maps cleanly onto Phase 7's RLS claims. NOT per-route inline checks; NOT the Better-Auth admin/access-control plugin.
- **D-12:** **Matrix + real gates.** Define the full `owner`/`practitioner`/`client` → capability matrix in code (constants), and **enforce the gates that are real today**: owner/practitioner-only admin surfaces (invite panel, privileged settings), client read-only. Structure data-scoping hooks so Phase 7 fills in subject-level enforcement. Don't wire untestable client/practitioner data rules into every loader yet.
- **D-13:** **Practitioner→client = tenant-scoped interim.** A practitioner may access all subjects within their own tenant for now. Per-assignment scoping (**AUTH-03**) and DB-level isolation (**RLS**, TEN-02/03) remain **Phase 7**.

### Claude's Discretion
- Dropdown/menu component implementation; settings page route path (`/account` vs `/settings`) and its sub-route structure; invite token format + exact expiry window; whether `invites` lives in `db/schema.ts` (tenancy domain) or `db/auth-schema.ts`; whether to keep `OWNER_INVITE_TOKEN` as a bootstrap fallback; role-aware nav rendering (hide-in-UI is cosmetic — the loader gate is authoritative).

</decisions>

## Existing Code Insights

### Reusable Assets
- `_app/layout.tsx` already returns `{ user: session.user }` (role included) — the shell + authz helpers consume this; no new session plumbing needed at the gate.
- `/logout` action exists and correctly forwards Better-Auth's cookie-clearing `Set-Cookie` (CR-01) — the account menu only needs a `Form` POST to it.
- Better-Auth `beforeSignUp` middleware + `changePassword` API + role `additionalFields` (`input:false`) are already configured — invites and password-change build on them.
- Phase 4.1 brand design system + `UI-SPEC.md` + `app/components/ui/` — the account menu, settings page, and invite UI must be in-brand and typed.

- Explicit route table in `app/routes.ts` (`RouteConfig`: index/route/layout) with all app routes under one authenticated parent `layout()` — add settings there; gate admin surfaces with the new authz helpers inside loaders/actions.
- Drizzle migrations baseline (Phase 1) — the `invites` table ships via `npm run db:generate` + `db:migrate` against Neon project `orange-paper-97068012`.
- `TopNav` uses CSS-var brand tokens + `NavLink`; `ThemeToggle` already lives top-right — the avatar menu joins/replaces that cluster.

### Integration Points
- Neon project `orange-paper-97068012` — the `invites` migration lands here (still Neon; Supabase parked).
- `OWNER_INVITE_TOKEN` env (Vercel prod+preview) — superseded by the `invites` table (D-07); coordinate any removal with the existing seed/bootstrap path.

- **Supabase migration (full re-platform: Auth + Postgres + Storage + RLS)** — **project-level decision, parked.** The owner floated it (motivated by built-in email invites + consolidated RLS/storage) but chose **Pause** after the blast radius surfaced. A migration would supersede the just-shipped Phase 3 (Better-Auth/Neon) + the Phase 1 spike and rewrite Phases 4/5/7. **Revisit deliberately before Phase 7** (its natural payoff point), and only once a Supabase project is actually provisioned — ideally de-risked via a timeboxed `/gsd-spike`.
