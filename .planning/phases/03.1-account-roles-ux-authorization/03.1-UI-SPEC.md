## Spacing Scale

Identical to Phase 4.1 — Fibonacci-derived DS tokens. Every Phase 3.1–specific measure
below resolves to an existing DS token (4 / 8 / 12 / 16 / 24 / 32 / 48); no off-scale
magic numbers are introduced, so the "no new spacing exceptions" claim holds true.
`--gap-md` (12px) is an inherited Phase 4.1 token (see the Typography "Inherited
Design-System Deviation" note), not a new exception. The two layout dimensions that are
not spacing tokens — the dropdown panel width (200px) and the settings column width
(680px) — are container sizing measures, not spacing-scale values, and are specified as
such (they govern content width, never inter-element rhythm).

### Everyday gap aliases (use these in Tailwind utilities)

| Token | Value | Tailwind utility | Usage |
|-------|-------|-----------------|-------|
| `--gap-xs` | 4px | `gap-1` | Icon-label gap, badge internal padding, badge icon gap |
| `--gap-sm` | 8px | `gap-2` | Compact element spacing, menu item gap, trigger gap |
| `--gap-md` | 12px | `gap-3` | Form field internal gap, menu item horizontal padding, trigger horizontal padding |
| `--gap-lg` | 16px | `gap-4` | Default element spacing |
| `--gap-xl` | 24px | `gap-6` | Section padding, card internal |
| `--gap-2xl` | 32px | `gap-8` | Layout gaps, page horizontal padding |
| `--gap-3xl` | 48px | `gap-12` | Major section breaks, settings page top padding |

### Phase 3.1–specific layout measures

All values below resolve to a gap token (4/8/12/16/24/32/48) or are container-width
measures (200px dropdown / 680px settings column).

| Element | Value | Resolves to | Rationale |
|---------|-------|-------------|-----------|
| Account dropdown menu width | 200px | container width (not a spacing token) | Fits "Account settings" + 16px icon in one line at 16px body |
| Settings page max-width | 680px centered in the 1180 shell | container width (not a spacing token) | Settings is a form hub — not a data-dense screen |
| Settings section card padding | 24px | `--gap-xl` | Comfortable form density (matches `Card` padding="lg") |
| Settings section gap between sections | 32px | `--gap-2xl` | Visual separation between functional groups |
| Invite token URL display box padding | 8px 12px | `--gap-sm` / `--gap-md` | Matches the `Input` md vertical rhythm (`Input` md is 44px height with 0 14px horizontal) — the box reuses the brand control padding token rather than inventing an ad-hoc value, and aligns optically against adjacent `Input` fields |
| Invite token URL display box radius | `--radius-md` | inherited radius token | Matches `Input` md radius |
| Touch targets (avatar trigger, menu items) | min 44px height | — | WCAG 2.5.5, matches existing BottomTab pattern |

## Typography

All sizes and weights are inherited from Phase 4.1 — no new sizes or weights are introduced.

| Role | Token | Size | Family | Weight | Line-height | Usage in 3.1 |
|------|-------|------|--------|--------|-------------|--------------|
| Micro label / eyebrow | `--text-2xs` | 11px | Mono | 400 | — | Section eyebrows in settings (PROFILE, SECURITY, INVITES, PREFERENCES, COMING SOON), invite role badge label, menu role chip |
| Meta / hint | `--text-xs` | 12px | Mono | 400 | — | Input hint text, invite expiry countdown, token hash display, invite created-at timestamp |
| Secondary body | `--text-sm` | 14px | Text | 400–500 | 1.5 | Form field labels, settings description text, menu item sub-labels, invite table cell copy |
| Body (primary) | `--text-base` | 16px | Text | 400 | 1.5 | Dropdown menu items, settings body text, invite URL display |
| Card title / section heading | `--text-lg` | 22px | Display | 500–600 | 1.18 | Settings section card headings ("Profile", "Security", "Invites", "Preferences") |
| Page title | `--text-2xl` | 36px | Display | 600 | 1.04–1.18 | Settings `PageHeader` title ("Account") |

**No new sizes.** Use only the values above. The `--text-md` (18px) size is available for the settings page subtitle / lead copy if needed, matching Phase 4.1 precedent.

### Inherited Design-System Deviation (justified)

This phase uses **6 sizes and 3 weights** (400 regular, 500 medium, 600 semibold). That is more than the generic per-phase "max 4 sizes / 2 weights" UI heuristic — and that is intentional and correct here:

- These 6 sizes / 3 weights are a strict **subset** of the approved Phase 4.1 type scale (the design-system-of-record: 9 sizes across `--text-2xs`…`--text-3xl`, 4 weights 400/500/600/700, three font families). Source: `04.1-UI-SPEC.md` §Typography (status: approved 2026-06-07).
- This is a **locked inheritance**, not ad-hoc sprawl. Phase 4.1 exists precisely to enforce a single binding type system across the whole app; collapsing the account/settings type scale to 4 sizes / 2 weights would make these screens inconsistent with every other shipped Zoetrop surface — the opposite of the design-system mandate.
- This inherited deviation therefore **overrides the generic per-phase 4-size/2-weight heuristic** for consistency with the shipped app.
- **No NEW sizes or weights are introduced by this phase** — every value is already present in the approved Phase 4.1 contract. The `--gap-md` (12px) spacing token is part of the same inherited DS-of-record and is likewise not a new exception.

Source: Phase 4.1 UI-SPEC §Typography (approved design-system-of-record).

## Interaction Contracts

### Dropdown open/close

```
Trigger click → open (aria-expanded: true)
Outside click → close
Escape key → close + return focus to trigger
Tab through items → closes on last item tab-out
Keyboard navigation: ArrowDown/ArrowUp moves focus between menu items
Focus visible: --shadow-ring on trigger and items
Animation: opacity 0→1 + translateY(-4px)→0, --dur-fast (120ms) --ease-out
```

### Invite copy flow

```
1. User clicks "Generate invite link"
2. POST /settings/invites (action) → returns { token, url, role, expiresAt }
3. Link display card appears (no page navigation)
4. User clicks "Copy link"
5. navigator.clipboard.writeText(url)
6. Copy icon → Check icon (--accent) for 1500ms → reverts
7. Card persists until dismissed (× button) or page reload
8. Invite appears in "Active invites" table immediately (optimistic or after POST confirm)
```

### Revoke flow

```
1. User clicks "Revoke invite" in invite row
2. Inline confirmation appears within the same row:
   "Revoke this invite? This cannot be undone. [Keep invite] [Revoke invite]"
   (NOT a modal — inline row expansion to avoid interaction cost)
3. Keep invite: collapses, no action
4. Revoke invite: POST /settings/invites/{id}/revoke → row status → REVOKED
5. Row remains visible with REVOKED badge (does not disappear — audit feel)
```

### Password change flow

```
Current password → New password → Confirm new password → "Save password" Button(primary)
On success: inline green message (--vital-400): "Password updated."
On error (wrong current password): Input error on "Current password" field:
  "Current password is incorrect."
On mismatch: Input error on "Confirm new password":
  "Passwords do not match."
On weak password: Input error on "New password":
  "Password must be at least 8 characters."
```

### Theme toggle in settings

```
Switch(tone: "focus", size: "md") + label "Dark mode"
Mirrors the TopNav ThemeToggle behavior (same localStorage key: "zt-theme")
Changing this switch updates the theme immediately (same code path as ThemeToggle)
```

## Dark Mode

Full dark mode support is inherited from Phase 4.1. All new components use CSS var tokens exclusively — no hardcoded hex. The warm-dark remap in `app.css` `html[data-theme="dark"]` block applies automatically.

Phase 3.1–specific notes:
