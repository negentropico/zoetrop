<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → app routes | Every /dashboard, /metrics, /protocol, /insights, /import request must carry a valid session — enforced in one layout loader |
| public surface ↔ private surface | landing/login/logout/api.auth are public; everything under _app/ is gated — the split must be airtight |
| Vercel env → deployed app | A lingering PILOT_BASIC_AUTH value could re-enable the stopgap — deleted as part of D-05 |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-unauth-access | Information disclosure | all app routes | mitigate | Authenticated layout loader `throw redirect("/login")` when no session — not opt-in per route; verified by tests/routes/auth-layout.test.ts + the manual /dashboard-while-signed-out check |
| T-03-pilot-leak | Information disclosure | PILOT_BASIC_AUTH env var | mitigate | root.tsx loader + headers() deleted (Task 2); the Vercel env var deleted (Task 3) so a leaked value cannot re-gate/expose the pilot |
| T-03-public-shell | Information disclosure | AppShell on public routes | mitigate | AppShell moved to _app/layout.tsx; root renders a bare Outlet so landing/login never inherit the authenticated shell (Pitfall 3) |
| T-03-session | Spoofing | session cookie on protected routes | mitigate | getSession validates the signed cookie server-side every request (cookieCache 5min); redirect on invalid/missing |
</threat_model>
