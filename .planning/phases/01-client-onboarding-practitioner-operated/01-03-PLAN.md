| caller → createSubject | tenantId + intake fields come from a server action that has already passed requireRole; createSubject itself trusts its caller but always scopes writes to the passed tenantId |
| checklist read → RLS | PHI reads run inside withTenantDb (SET LOCAL GUC + app_user role); the invites read uses admin path but filters tenantId + subjectId explicitly |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-01-checklist-bleed | Information Disclosure | getChecklistStatus queries | mitigate | Every query filters BOTH tenantId AND targetSubjectId; PHI reads additionally RLS-governed via withTenantDb (Task 2, D-11 defense-in-depth) |
| T-01-subject-write-scope | Tampering | createSubject | mitigate | Write always carries the caller-supplied tenantId; the calling route (Plan 06) gates with requireRole owner/practitioner before invoking (Task 1) |
| T-01-server-leak | Tampering | subjects/checklist client-bundle leak | mitigate | Both modules keep .server.ts suffix; consumed only by server loaders/actions; build gate enforces (Task 2 verify runs npm run build) |
| T-01-false-done | Repudiation | checklist labs/genetics state | mitigate | Labs require status='approved' (not mere presence); honest 3-state prevents a misleading "done" while extractions await review (Task 2, D-10) |
