<objective>
Build the active-subject scoping engine (ONB-03, D-04/D-06): a `getActiveSubject` resolver that reads a session-scoped httpOnly `zt-subject` cookie, the `/subject-switch` resource route that sets it, the single `requireSubjectCtx` swap that re-scopes all 13 Pattern-A PHI loaders, and the 7 manual Pattern-B ingest/report swaps.

Purpose: This is the PHI-scoping spine. One resolver swap makes every metrics/protocol/insights/dashboard loader scope to whichever subject the practitioner has selected, and the ingest/report swaps let a practitioner upload + review + report on a client subject. The cookie is httpOnly + session-scoped so the practitioner never resumes "inside a client" unexpectedly.
Output: `getActiveSubject` in data.server.ts, `subject-switch.ts` route, `requireSubjectCtx` swap, 7 Pattern-B loader swaps, routes.ts registration.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → /subject-switch | Untrusted form-supplied subjectId crosses here; must be validated against DB + tenant + assertSubjectAccess before the cookie is set |
| browser cookie → getActiveSubject | The zt-subject cookie value is attacker-controllable; every read re-validates it against the tenant |
| client JS → zt-subject cookie | httpOnly blocks client JS read/write; only the server action sets it |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-cookie-tamper | Tampering | zt-subject cookie | mitigate | httpOnly (no client JS write); getActiveSubject re-validates the value against subjects WHERE tenantId on every request; self-heals to owner fallback on invalid value (Task 1) |
| T-01-cross-tenant | Information Disclosure | subject-switch action + getActiveSubject | mitigate | assertSubjectAccess Gate 2 (cross-tenant→403) in the action; getActiveSubject SELECT filters `tenantId = user.tenantId` so a cross-tenant cookie never resolves (Task 1, Task 2) |
| T-01-client-403 | Elevation of Privilege | subject-switch action + requireSubjectCtx | mitigate | assertSubjectAccess Gate 1 (client→403) fires in the switch action AND unchanged in requireSubjectCtx on every PHI loader; client role can neither switch nor read (Task 2) |
| T-01-session-scope | Session Management | zt-subject cookie | mitigate | No Max-Age/Expires → session-scoped; closing the browser resets to owner (D-06, Pitfall 5, Task 2) |
| T-01-patternB-leak | Information Disclosure | ingest/report loaders | mitigate | Pattern-B swap ensures uploads/reviews/reports scope to the active subject, not silently to the owner (Task 3, Pitfall 3) |
| T-01-server-leak | Tampering | data.server.ts client-bundle leak | mitigate | getActiveSubject stays in .server.ts; SubjectChip (Plan 05) consumes via loader props only; build gate enforces (Task 3 verify runs npm run build) |
</threat_model>
