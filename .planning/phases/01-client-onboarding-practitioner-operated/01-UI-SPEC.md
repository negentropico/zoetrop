## Spacing Scale

**IMPORTANT: This project uses a Fibonacci-derived (non-linear) modular scale, NOT a multiple-of-4 scale.** The scale is intentionally non-linear — do not "correct" it to 4/8/16/24 multiples.

| CSS Token | rem | px (approx) | Everyday alias | Usage |
|-----------|-----|-------------|----------------|-------|
| `--space-1` | 0.125rem | 2px | — | Fine-grained optical spacing |
| `--space-2` | 0.25rem | 4px | `--gap-xs` | Icon internal gaps |
| `--space-3` | 0.5rem | 8px | `--gap-sm` | Tight row padding, badge internals |
| `--space-4` | 0.8125rem | 13px | — | Transitional gap |
| `--space-5` | 1.3125rem | 21px | — | Transitional gap |
| `--space-6` | 2.125rem | 34px | — | Section-level separation |
| `--space-7` | 3.4375rem | 55px | — | Large layout padding |
| `--space-8` | 5.5625rem | 89px | — | Hero-level spacing |

**Preferred everyday aliases (use these in implementation, not raw space-* tokens):**

| Alias | Value | Usage |
|-------|-------|-------|
| `--gap-xs` | 0.25rem / 4px | Icon gaps, inline tight spacing |
| `--gap-sm` | 0.5rem / 8px | Badge padding, element internal gaps |

## Color

> Tokens live in `app.css` Layer 1/2. All color references use CSS var strings — dark mode remaps automatically via `html[data-theme="dark"]`.

### 60 / 30 / 10 Contract

| Role | Token | Light value | Usage |
|------|-------|-------------|-------|
| Dominant surface (60%) | `--bg` → `var(--paper)` | `#f5f2f0` | Page canvas, entire app background |
| Secondary surface (30%) | `--surface` → `var(--n-0)` | `#ffffff` | Cards (`.zt-card` / `Card`), sidebar footer popover, form panels |
| Secondary (low) | `--surface-low` → `var(--n-100)` | `#ece7e6` | Nav rail, sidebar chrome |
| Accent (10%) | `--accent` → `var(--focus-300)` | `#8070ff` | **Reserved-for list below** |
| Destructive | `--danger` | `#e5484d` | Revoke invite, destructive actions only |

**Accent `--accent` (#8070ff periwinkle) is RESERVED FOR:**
1. `Button variant="primary"` fill — the "Add client" primary CTA and "Generate invite" CTA
2. Active-subject chip ONLY when a non-owner client subject is selected (the visual danger signal)
3. Focus ring on any interactive element (via `--shadow-ring`)
4. Text links within copy (`.zt-link`)

**Accent is NOT used for:** status indicators, general informational badges, nav highlights, checklist states.

### Status colors (checklist 3-state)

These existing status tokens map to the checklist states for this phase:

| Checklist State | Token | Light value | Glyph |
|-----------------|-------|-------------|-------|
| Done (approved/present) | `--optimal` → `var(--vital-400)` | `#0aa183` (teal) | `✓` |
| In progress (data exists, not approved) | `--borderline` → `var(--energy-400)` | `#f0ac1f` (amber) | `·` (mid-dot) |
| Missing (no data) | `--text-faint` → `var(--n-400)` | `#9c9295` (muted warm grey) | `∘` (ring) |

> **NOT color-only:** each state also has a distinct glyph (`✓` / `·` / `∘`) and a text label for screen reader and reduced-contrast accessibility.

### Subject-chip elevated state (PHI-safety affordance)

When viewing a non-owner client subject (D-05/D-06), the chip shifts from its resting neutral state to a visually distinct **accent-tinted** state:

| Chip state | Background | Text | Border |
|------------|------------|------|--------|
| Resting (viewing owner) | `var(--surface-sunken)` | `var(--text-secondary)` | none |
| Elevated (viewing a client) | `var(--focus-50)` (#efecff) | `var(--accent)` (#8070ff) | `1px solid var(--accent)` + left-side 2px ink bar |

The elevated state uses the same `--focus-50` wash the `zt-docline.is-hot` and `zt-consent-row.is-on` patterns use — consistent visual language for "active/selected" within the design system.

## Surface Specifications

### Surface 1 — Active-Subject Identity Chip (D-05/D-06)

**Location:** Sidebar footer (`remix-app/app/components/shell/`), inserted ABOVE `SidebarAccount` in the `.zn-foot` block.

**Resting state (viewing owner):**
```
[ ○ Owner Name  ▾ ]          ← 30px avatar, "Owner Name", ChevronDown icon
  VIEWING                     ← zt-eyebrow subline, text-muted
```
- Container: width 100%, `padding: 7px 8px`, `border-radius: var(--radius-md)`
- Background: `var(--surface-sunken)` — same resting treatment as `.zn-account`
- Avatar: `Avatar name={subject.displayName} size={30}`
- Label: `font-size: var(--text-sm)`, `font-weight: 600`, `color: var(--text)`
- Subline: `VIEWING` in `.zt-eyebrow` style (font-mono, 11px, uppercase, text-muted)
- Chevron: `ChevronDown` from lucide-react, size=15, `color: var(--text-muted)`, `margin-left: auto`

**Elevated state (viewing a non-owner client subject):**
- Background: `var(--focus-50)` (#efecff light / dark-remapped)
- Border: `1px solid var(--accent)` + left accent bar `2px solid var(--accent)` (absolute, inset left)
- Label color: `var(--accent)` (#8070ff)
- Subline: `VIEWING CLIENT` (not just "VIEWING") — makes the distinction unmistakable
- The visual change must be immediate on cookie-switch redirect

**Dropdown (open state):**
- Same `.zn-account-menu` popover CSS — opens upward (`bottom: calc(100% + 8px)`)
- In rail/collapsed mode: opens to the right (`.is-rail` class)
- Popover header: mono eyebrow "SWITCH SUBJECT"
- List: owner subject first, then client subjects in alphabetical order
- Each row: avatar (24px) + display name + role badge (`Badge tone="neutral" variant="soft"` for owner, `Badge tone="focus" variant="soft"` for client)
- Active subject: `font-weight: 600`, `background: var(--surface-sunken)`, checkmark icon right
- Interaction: clicking a row submits the hidden form → POST `/subject-switch` → server sets `zt-subject` cookie → redirect back to same page
- No client JS cookie writes — form submit only
- Escape key closes

**Collapsed rail state:**
- Show Avatar only (no text) — same as `.zn-account` rail treatment
- Popover opens to the right, width 224px
- Elevated-state visual: avatar gets a 2px periwinkle ring (`box-shadow: 0 0 0 2px var(--accent)`)

**Accessibility:**
- `role="menu"` on dropdown, `role="menuitem"` on each row
- `aria-expanded` on trigger
- `aria-label="Switch active subject"` on trigger button

### Surface 2 — `/clients` Page (D-11)

**Route:** `/clients` (new, registered in `routes.ts`)

**Layout:** Single-column within `.zn-page` max-width 1180px, using `PageHeader` + sections with `--gap-section` vertical rhythm.

**Page Header:**
```
CLIENTS                    [eyebrow]
Clients        [+ Add client]  [right slot]
```
- `<PageHeader eyebrow="CLIENTS" title="Clients" right={<Button variant="primary" iconLeft={<Plus />}>Add client</Button>} />`
- Crumb: auto-derived (single segment, suppressed per existing rule)

**Client list (populated state):**
- Layout: `Card elevation="sm" padding="none"` containing an inline `<table>` with `.zt-trow` rows
- Table headers (`.zt-eyebrow` style, font-mono, 11px): NAME / PROGRAM / STATUS / ONBOARDING / ACTIONS
- Each row columns:
  1. **NAME**: `Avatar size={24}` + display name (`font-weight: 600, font-size: var(--text-sm)`) + contact email below in `text-muted text-2xs` (font-mono)
  2. **PROGRAM**: program type badge (`Badge tone="neutral" variant="soft"`) + program start date in `font-mono text-2xs text-muted`
  3. **STATUS**: invite status (`Badge` — "Pending" neutral/soft, "Active" vital/soft, "Not sent" neutral/outline)
  4. **ONBOARDING**: inline checklist strip (see Checklist Strip spec below)
  5. **ACTIONS**: `Button variant="ghost" size="sm"` "View" → navigates to client detail + switches active subject

**Checklist strip (at-a-glance row form):**
```
intake✓  gen·  lab∘  whp✓  report∘  protocol∘
```
- Each segment: `StatusDot size={7}` + mono abbreviation (`font-mono text-2xs`)
- States: ✓ done (teal `--optimal`), · in-progress (amber `--borderline`), ∘ missing (muted grey `--text-faint`)
- Full labels (screen reader `aria-label`): "Intake: done", "Genetics: in progress", "Labs: missing", "WHOOP: done", "Report: missing", "Protocol: missing"
- Rendered as `<span aria-label="...">` sequence in a `display: flex; gap: var(--gap-sm); align-items: center` container

**Empty state (no clients yet):**
- Center of card area (padding `var(--gap-3xl)` top/bottom)
- Eyebrow: `NO CLIENTS YET` (`.zt-eyebrow`)
- Heading: `Add your first client` (font-display, text-lg, fw-600)
- Body: `Create a client record to begin. You can enter data before they redeem their invite.` (text-sm, text-muted, max-width 400px)
- CTA: `<Button variant="primary" iconLeft={<Plus />}>Add client</Button>`

**Expanded per-client checklist (client detail or slide-over):**

A vertical checklist showing each step with its 3-state indicator:

| Step | Label | "Done" condition |
|------|-------|-----------------|
| Intake | Intake & consent | `subjects` has dob + biologicalSex + `consent_log` row |
| Genetics | Genetics | `subject_genotypes` count > 0 |
| Labs | Lab results | `lab_extractions` has `status = 'approved'` row |
| WHOOP | WHOOP data | `metrics` has `source = 'whoop'` row |
| Report | Report | `reports` row exists |
| Protocol v1 | Protocol v1 | `protocol_versions` row exists |
| Account | Account (invite) | Informational: not-sent / sent / redeemed |

- Each row: `StatusDot size={9}` + label (text-sm) + detail text (text-2xs text-muted, font-mono, e.g. "Approved 2026-06-14" or "2 extractions pending review")
- Account row uses `Badge` not `StatusDot` — it is informational, not a gating step

### Surface 3 — Create-Client Intake Form (D-07/D-08)

**Trigger:** Clicking "Add client" on `/clients` opens the form. Implementation: either a slide-over panel (preferred for practitioner workflow continuity) or a dedicated `/clients/new` route. The UI contract applies to either; layout is single-column, max-width 560px.

**Form sections and fields:**

**Section 1: Identity** (eyebrow: `CLIENT IDENTITY`)
- `displayName`: text input, label "Full name", required, placeholder "First Last"
- `dob`: date input, label "Date of birth", required
- `biologicalSex`: `<select>` styled per `assignments.tsx` pattern, label "Biological sex", options: Male / Female / Intersex, required
  - Note: "biological sex" not "gender" — this drives clinical reference-range logic

**Section 2: Contact** (eyebrow: `CONTACT`)
- `contactEmail`: email input, label "Email address", optional, placeholder "client@example.com"
- `contactPhone`: tel input, label "Phone (optional)", optional

**Section 3: Program** (eyebrow: `PROGRAM`)
- `programType`: `<select>`, label "Program type", options: Cessation / Substance taper / Lifestyle modification / General, required
- `programStartDate`: date input, label "Program start date", optional

**Section 4: Intake notes** (eyebrow: `INTAKE NOTES`)
- `goals`: `<textarea>`, label "Client goals", optional, rows=3, placeholder "What is the client aiming to achieve?"
- `intakeNotes`: `<textarea>`, label "Practitioner notes", optional, rows=3, placeholder "Clinical context, history, flags for the protocol…"

**Form actions:**
- Primary: `<Button variant="primary" type="submit">Create client</Button>`
- Secondary: `<Button variant="ghost" type="button">Discard</Button>`
- Inline validation: required fields show `color: var(--danger)` error below the field on blur/submit attempt

**Post-submit behavior:** On success, redirect to `/clients` with the new client row visible and active subject cookie left unchanged (remains owner until practitioner explicitly switches).

### Surface 4 — Invite Generation (settings/invites pattern, extended)

Invite generation lives within the `/clients` detail area (or as a card on `/clients`). Reuses the existing `settings/invites.ts` resource route pattern; the UI contract adds the "shown once" token reveal affordance.

**Generate invite flow:**
1. "Generate invite link" button (`Button variant="secondary"`, inside client card)
2. On action success, reveal panel appears inline:
   - Background: `var(--focus-50)`, border `1px solid var(--accent)`, `border-radius: var(--radius-lg)`, padding `var(--gap-lg)`
   - Eyebrow: `INVITE LINK — SHOWN ONCE` (font-mono, 11px, uppercase, `color: var(--accent)`)
   - Token display: monospaced code block, `font-mono`, `font-size: var(--text-2xs)`, `background: var(--surface)`, `border: 1px solid var(--border)`, `border-radius: var(--radius-sm)`, `padding: var(--gap-sm) var(--gap-md)`, full-width, selectable
   - Copy button: `<IconButton>` (Copy lucide icon) right of token, pressing it copies raw token and switches icon to `Check` for 2 seconds
   - Warning copy: `Copy this token now — it will not be shown again. Deliver it to your client out of band (email, message, etc.).`
   - Font: text-sm, color: `var(--text-secondary)`
3. After navigation away, token display is gone; invite row shows status "Sent — pending redemption"

**Invite status row (in client checklist):**
- Not sent: `Badge tone="neutral" variant="outline"` "No invite" + "Generate invite link" action
- Sent (pending): `Badge tone="energy" variant="soft"` "Pending" + `font-mono text-2xs text-muted` expiry date + "Revoke" ghost button
- Redeemed: `Badge tone="vital" variant="soft"` "Active" + `font-mono text-2xs text-muted` redemption date

## Interaction Contracts

### Subject switch flow
1. Practitioner opens chip dropdown
2. Selects a subject row → form submit (`POST /subject-switch`)
3. Server validates via `assertSubjectAccess`, sets `zt-subject` httpOnly session cookie
4. Redirect to same page (server-computed `Referer` header)
5. Chip re-renders in elevated state (focus-50 wash + accent border)
6. All PHI loaders now scope to the selected subject
7. **Duration:** `--dur-base` (200ms) `--ease-out` for chip background transition

### Create client form submit
1. Practitioner fills form → "Create client" submit
2. Server action validates required fields server-side (not trust client)
3. On error: return action data with field errors, re-render form with inline error messages below each field
4. On success: redirect to `/clients` — new row visible in list, active subject unchanged (remains owner)

### Invite token copy
1. Generate invite → server action returns raw token (one time)
2. Token reveal panel appears with copy button
3. Click copy → `navigator.clipboard.writeText(token)` → button icon changes Copy → Check for 2s → reverts
4. Navigating away from page makes token inaccessible — no re-fetch

### Checklist polling / refresh
- Checklist state is read at page-load time (Remix loader). No real-time polling.
- Practitioner refreshes the page to see updated checklist state after ingest operations (Phase 2).
- "Done" states for genetics/labs require the review-approve gate to be cleared before showing.
