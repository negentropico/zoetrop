<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| chip form → /subject-switch | The chip only POSTs subjectId; the resource route (Plan 02) validates it via assertSubjectAccess before setting the cookie — the chip is not a trust boundary by itself |
| layout loader → chip props | Subject list crosses from server loader to client component as plain props (no .server import in the chip) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-chip-cookie-write | Tampering | SubjectChip | mitigate | Chip uses a server-POST form, never document.cookie; httpOnly cookie cannot be set client-side (Task 1, Pitfall 2) |
| T-01-server-leak | Tampering | SubjectChip client bundle | mitigate | Chip imports no .server module; subject list arrives via loader props; build gate enforces (Task 1 verify runs npm run build) |
| T-01-phi-mis-edit | Information Disclosure | active-subject ambiguity | mitigate | Persistent elevated chip ("VIEWING CLIENT" + accent wash) makes operating on a client unmistakable (D-05 PHI-safety affordance, Task 1) |
| T-01-misleading-state | Repudiation | cessation hero for non-cessation subject | mitigate | hasCessationProgram guard replaces a fake "Day 0 · Acute" with an honest placeholder (Task 2, Pitfall 6) |
| T-01-chip-list-leak | Information Disclosure | layout subject list | mitigate | listSubjectsForTenant is tenant-scoped; the chip list is only populated for owner/practitioner roles (clients get an empty list) (Task 1) |
</threat_model>
