<objective>
Resolve the single highest-uncertainty M1 unknown before Phase 3 builds on it: can a Better-Auth-issued JWT be verified by Neon's `pg_session_jwt`, with `tenantId`/`subjectId` claims readable via `current_setting()`/`auth.session()` inside a Postgres transaction, and does a throwaway RLS policy keyed on the claim correctly flip row visibility? The spike also MUST demonstrate the `SET LOCAL` vs bare `SET` distinction — that tenant context does not leak across pooled connections (D-05). The deliverable is a VERDICT + recommendation in `01-SPIKE-FINDINGS.md`, not the real tenancy spine.

This is thin, timeboxed (~1 day), and throwaway (D-03). It runs ONLY against a DISPOSABLE Neon branch with no production PHI. If JWK verification cannot be made to work in the timebox, the spike fails closed to the `SET LOCAL app.tenant_id` pattern (D-04) and records that as the verdict. Spike code lives under `remix-app/spikes/` (never under `app/`), and the `spike-*.ts`/`.sql` files are deleted after the findings note is committed (D-03, Pitfall 6).

Purpose: De-risk Phase 3 (Identity + Tenancy Spine with RLS) so it builds against a proven seam, not an assumed one.
Output: Throwaway spike harness under `remix-app/spikes/auth-rls/`, executed against a disposable Neon branch, and a committed `01-SPIKE-FINDINGS.md` verdict.
</objective>
