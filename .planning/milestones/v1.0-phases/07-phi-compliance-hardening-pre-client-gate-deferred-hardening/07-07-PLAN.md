<objective>
Close the two access-control gaps that prevent AUTH-03 from being achieved at runtime on the
PHI-read surfaces, and fix the one warning that falsifies the immutable audit log.

This plan is the pure-code half of the second gap-closure round. It does NOT touch the schema or
the database — it wires the already-verified per-assignment Gate-3 pattern into the two routes that
were missed in plan 07-05, and corrects the hardcoded audit role.

Blocker (CR-01): `document.tsx` (GET /ingest/documents/:id) — the raw lab PDF byte stream, the
highest PHI-density artifact in the system — calls `assertSubjectAccess` with only 3 arguments and
no `subject.id`. Gate 3 is permanently skipped, so an unassigned practitioner in the tenant can
fetch any subject's raw lab PDF. The `review.tsx` loader that renders this PDF IS gated, but the PDF
URL it embeds is freely accessible. This plan wires the same `listAssignedSubjectIds` →
`assertSubjectAccess(…, assignedIds)` pattern that 5 sibling routes already use (review.tsx,
upload.tsx, generate.tsx, index.tsx, detail.tsx).

Warning folded in (same fix shape): `consent.tsx` loader (line 34) AND action (line 55) use the
same 3-arg `assertSubjectAccess`, letting an unassigned practitioner record a consent record (a
legal artifact) for a subject they are not authorized for.

Warning folded in (WR-04): `audit.server.ts` `insertAuthAuditLog` hardcodes `role: 'owner'`, so
every practitioner/client sign-in row in the immutable audit log is falsified. The actual role is
available from the user row already queried in `auth.server.ts`; thread it through.

Purpose: an unassigned practitioner CANNOT fetch the raw PDF and CANNOT record consent for an
unassigned subject; auth audit rows carry the real actor role.
Output: 2 hardened route files, 1 corrected audit writer + its 3 call sites, extended wiring tests.
</objective>
