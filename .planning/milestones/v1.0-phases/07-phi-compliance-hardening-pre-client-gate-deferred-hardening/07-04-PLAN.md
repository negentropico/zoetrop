<objective>
Close AUTH-03 and AUTH-04 on top of the live RLS layer: extend `assertSubjectAccess` so practitioners pass only for their assigned subjects (owner keeps tenant-wide access), add an `assignments.server.ts` service + a minimal owner-facing `/settings/assignments` assign/unassign UI backed by the `practitioner_subject_assignments` table, and wire Better-Auth auth events (sign-in/out/redemption) into the immutable `audit_log`.

Purpose: AUTH-03 (a practitioner accesses only subjects assigned to them) and AUTH-04 (auth + access events written to an immutable audit log). The assignment table + RLS policy already exist (Plans 01/02); this plan adds the authz enforcement, the management surface, and the auth-event trail.
Output: per-assignment `assertSubjectAccess`, assignment service + owner UI under /settings, auth-event audit writes, and a fail-closed assignments unit test.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Practitioner → subject data | A practitioner must reach only assigned subjects; an unassigned subject is a privilege-escalation target |
| Owner assignment action → psa table | Only the owner may create/revoke assignments; a practitioner/client crafting the POST must be denied |
| Auth hook → audit_log | Auth events write to the immutable log; a tampered or dropped event breaks the AUTH-04 trail |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-14 | Elevation of Privilege | Practitioner accesses unassigned subject | mitigate | assertSubjectAccess denies practitioners whose assignedSubjectIds (from listAssignedSubjectIds) lacks the subject; owner-only assignment management; psa RLS policy confines reads/writes to the tenant |
| T-07-15 | Tampering | Non-owner manipulates assignments | mitigate | /settings/assignments loader+action both requireRole(user, ["owner"]); assignSubject/unassignSubject are tenant-scoped via withTenantDb (psa RLS WITH CHECK on app.tenant_id) |
| T-07-16 | Repudiation | Auth events missing from the log | mitigate | sign-in/out/sign-up hooks write audit_log rows via the admin path; AUTH-04 immutability (INSERT+SELECT-only RLS from Plan 02) makes them append-only |
| T-07-17 | Denial of Service | Audit-logging error breaks auth | mitigate | Each auth-event write is best-effort (try/catch) — a logging failure never throws into Better-Auth's sign-in/out flow |
| T-07-18 | Information Disclosure | PHI leaks into auth-event rows | mitigate | insertAuthAuditLog carries only userId/action/tenantId/entityId (D-13); no clinical value/name/result field |
| T-07-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this plan; supply-chain surface unchanged |
</threat_model>
