## What Was Built

### Schema Changes
- `remix-app/db/auth-schema.ts`: Added nullable `tenantId text('tenant_id')` to the `user` table. Plain text (no `.references()`) to avoid a circular import with `schema.ts`. Logical FK documented in a comment.
- `remix-app/db/schema.ts`: Added `invites` pgTable with: `id`, `tokenHash` (UNIQUE — SHA-256 hex only, no raw token), `role` (appRoleEnum), `tenantId` (FK→tenants.id), `createdBy` (FK→user.id), `expiresAt`, `consumedAt`, `consumedBy` (nullable FK→user.id), `revokedAt`, `createdAt`. Two indexes: `idx_invites_tenant` and `idx_invites_token_hash`. `invitesRelations` (tenant + createdByUser one-relations). Imports `user` from `./auth-schema` for the FK references.

### Migration
- `migrations/0005_fantastic_deadpool.sql`: Hand-corrected SQL (see Deviations). Adds `ALTER TABLE "user" ADD COLUMN "tenant_id" text` (nullable) and `CREATE TABLE "invites"` with all columns, FKs, and indexes. Applied to Neon project `orange-paper-97068012`.

### Authorization Helpers
- `remix-app/app/lib/authz.server.ts` exports:
  - `requireUser(request)` — calls `auth.api.getSession`; throws `redirect(/login?redirect=...)` if no session
  - `requireRole(user, allowed)` — fail-closed; throws `Response(403)` if `!user.role` or role not in allowed
  - `assertSubjectAccess(user, subject, userTenantId)` — denies clients; denies cross-tenant access (D-13 tenant-scoped interim)
  - `CAPABILITIES` — `{ owner: ["invite:practitioner","invite:client","admin:settings"], practitioner: ["invite:client","admin:settings"], client: [] }`
  - `can(user, capability)` — reads CAPABILITIES with `?? "client"` fallback (fail-closed for unknown roles)

### Tests
- `tests/auth/authz.test.ts`: 16 tests, all passing. Covers requireRole (5 cases), assertSubjectAccess (4 cases), CAPABILITIES+can (7 cases). Every threat model case from T-031-AZ-1/2/3 is covered.

### Seed Script
- `scripts/seed-owner.ts`: Updated role-elevation UPDATE to also set `tenantId` from the seeded tenant, binding the owner user to their tenant. Script remains idempotent.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Drizzle snapshot drift caused 0005 migration to try re-adding existing columns**
- **Found during:** Task 2 — inspecting the generated `0005_fantastic_deadpool.sql`
- **Issue:** Migrations 0002–0004 were hand-written SQL files applied directly to Neon; the Drizzle kit snapshots (0000–0004) did not track those changes. When `db:generate` compared `schema.ts` (which has `tenant_id`/`subject_id` on 8 health tables) against the 0004 snapshot (which didn't know about them), it generated ALTER TABLE statements for columns that already exist in the live DB — applying would have failed with "column already exists".
- **Fix:** Hand-corrected `0005_fantastic_deadpool.sql` to include ONLY `ALTER TABLE "user" ADD COLUMN "tenant_id" text` and `CREATE TABLE "invites"`. Updated `0004_snapshot.json` by replacing it with the 0005 snapshot minus the invites table and user.tenant_id, so the snapshot now correctly reflects the live DB pre-0005 state. Future `db:generate` calls will produce clean diffs.
- **Files modified:** `migrations/0005_fantastic_deadpool.sql`, `migrations/meta/0004_snapshot.json`
- **Commit:** b8ef4dd
