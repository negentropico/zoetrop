## What Was Built

### `remix-app/app/routes.ts` (modified)
Registered two routes inside the authenticated `layout("routes/_app/layout.tsx", [...])` block so they inherit the session gate:
- `route("settings", "routes/_app/settings/index.tsx")`
- `route("settings/invites/:inviteId/revoke", "routes/_app/settings/invites.ts")`

### `remix-app/app/routes/_app/settings/index.tsx` (created — the account hub)
- **loader** — `requireUser(request)`; computes `canInviteClient` (gated on `can(user, "invite:client") && !!user.tenantId`) and `canInvitePractitioner`; loads `listInvites(user.tenantId!)` only when `canInviteClient`. Returns `{ user, invites, canInviteClient, canInvitePractitioner }`.
- **action** — `requireUser(request)` then dispatches on a hidden `_intent`:
  - `update-profile` → `auth.api.updateUser({ body: { name } })`; returns "Changes saved." / "Unable to save changes. Try again."
  - `change-password` → validates length ≥ 8, confirm-match; `auth.api.changePassword({ body: { currentPassword, newPassword } })`; maps the wrong-current-password error to "Current password is incorrect." (T-031-SET-3 — no escalation path, current password required)
  - `generate-invite` → `requireRole(user, ["owner","practitioner"])` FIRST (server gate, T-031-SET-1), then `generateInvite({ inviter: user, role })` (helper re-enforces per-role policy, D-10); builds the copy-link URL.
- **default component (`SettingsPage`)** — 680px centered column with 8 sections in order:
  1. PROFILE — Avatar(48) + read-only email + editable display name; "Save changes"
  2. SECURITY — current/new/confirm password inputs with per-field error mapping; "Save password"
  3. INVITES (only when `canInviteClient`) — role-selector pills (Practitioner shown only when `canInvitePractitioner`), "Generate invite link", generated-link card with Copy→Check transition (1.5s, `--accent`) + aria-live announcement, ACTIVE INVITES table (ROLE/STATUS/EXPIRES/CREATED/ACTION) with status derived from `consumedAt`/`revokedAt`/`expiresAt`, inline revoke confirmation
  4. PREFERENCES — `Switch(tone:"focus")` wired to the same `zt-theme` localStorage path as ThemeToggle
  5-8. INTEGRATIONS / DATA EXPORT / PRIVACY & CONSENT / UNITS & DISPLAY — flat sunken cards, "Coming in a future update.", no controls
- Raw token NEVER rendered in the active-invites table (T-031-SET-4); only the freshly generated token appears once in the link card.

### `remix-app/app/routes/_app/settings/invites.ts` (created — revoke resource route)
Action-only resource route (no loader, no default export): `requireUser` → `requireRole(user, ["owner","practitioner"])` (client 403'd even with a crafted POST, T-031-SET-2) → `revokeInvite({ id: params.inviteId!, tenantId: user.tenantId ?? "" })` (tenant-scoped IDOR guard — the helper filters `WHERE id AND tenantId`).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Server-only `authz.server` dragged into the client bundle**
- **Found during:** Task 4 human-verify (dev-server load of `/settings`)
- **Issue:** `can()` was imported from the server-only `~/lib/authz.server` and called inside the client-rendered `SettingsPage` component at three sites (selectedRole init, INVITES card gate, Practitioner pill gate). Vite's import-analysis pulled `authz.server` into the client bundle and threw `Server-only module referenced by client '~/lib/authz.server'`. `tsc --noEmit` and `npm run build` did NOT catch this — only the dev server's import-analysis does.
- **Fix:** Computed the capabilities in the loader (`canInviteClient` / `canInvitePractitioner` booleans) and passed them through loader data. The component now destructures the booleans from `useLoaderData` and contains ZERO references to `can`/`requireUser`/`requireRole`/`authz.server`. `loader`/`action` retain their `can`/`requireRole` usage (React Router strips those server exports from the client bundle).
- **Files modified:** `remix-app/app/routes/_app/settings/index.tsx`
- **Commit:** cf33126
- **Audit:** Grepped every other route importing a `*.server` module — `login.tsx`, `layout.tsx`, `api.auth.$.ts`, `logout.tsx`, `settings/invites.ts` are all clean (server symbols used only in loader/action, or resource routes with no default export). No other file needed the same fix.

### Cross-Plan Gap Closed (orchestrator)

**2. [Cross-plan] Invite links were not redeemable — no sign-up surface existed at `/login`**
- **Found during:** Task 4 human-verify (full invite redemption flow)
- **Issue:** This plan's generated invite URL points at `/login?inviteToken=...&role=...`, which the plans assumed was an existing sign-up surface. But `/login` only performed sign-in — there was no account-creation path consuming the token, so a copied invite link could not actually onboard a new user.
- **Fix (orchestrator, commit 9664108):** Enhanced `routes/auth/login.tsx` so that when an `inviteToken` is present, it renders a "Create your account" form that calls `auth.api.signUpEmail` with the token. Plan 02's `beforeSignUp` hook then assigns role + tenant from the invite row and marks it consumed. This is the piece that makes the invite links generated by this plan's INVITES panel redeemable end-to-end (generate → copy → open link → create account → role+tenant assigned → invite flips to USED).
- **Files modified:** `remix-app/app/routes/auth/login.tsx` (commit 9664108, outside this plan's file set)

### Data Repair (no commit) — known seed/data gap

**3. [Data] Owner `user.tenantId` was NULL — silently hid the INVITES section**
- **Found during:** Task 4 human-verify ("where do I create an invite?")
- **Issue:** The signed-in owner's `user.tenantId` was NULL: the owner account predated Plan 01's `tenantId` column, and `scripts/seed-owner.ts` early-exits on an existing user, so it never backfilled the column. Because the INVITES section is gated on `canInviteClient = can(user, "invite:client") && !!user.tenantId`, the section was silently hidden for the tenantless owner — the fail-closed gate (T-031-SET-5) behaved correctly but with no explanatory UI.
- **Repair (orchestrator, no commit):** Backfilled `owner.tenantId` → the existing "Owner Tenant" (`481b86b3-e029-4caa-8bab-a2d11d2e2a6a`). The user re-logged-in (5-minute session cookie-cache window) and the Invites section appeared.
- **Known gap (carry-forward):** `seed-owner`'s idempotent early-exit cannot backfill an existing owner's tenantId, and the settings UI silently hides invites when an owner has no tenant. Candidate future UX: an explanatory empty-state for the tenantless-owner case rather than a hidden section. Not actioned in this plan (no code change required for the approved flow).
