<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → /settings loader/action | Untrusted requests reach profile/password/invite mutations; every entry gated by requireUser + (for invites) requireRole |
| client → /settings/invites/:id/revoke | Resource route mutating invite state; must be role-gated AND tenant-scoped |
| changePassword | Account-takeover surface — must require the current password |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-031-SET-1 | Elevation of Privilege | generate-invite action | mitigate | `requireRole(user, ["owner","practitioner"])` BEFORE generateInvite; helper re-enforces per-role policy via `can`. UI hide is non-authoritative. |
| T-031-SET-2 | Elevation of Privilege | revoke resource route | mitigate | `requireRole` gate + `revokeInvite` scoped by `user.tenantId` (cannot revoke another tenant's invite — IDOR closed). |
| T-031-SET-3 | Spoofing / Account takeover | change-password | mitigate | `auth.api.changePassword` requires `currentPassword`; no reset-without-current path exposed; wrong current → generic "incorrect" error (no oracle). |
| T-031-SET-4 | Information Disclosure | invite token in table | mitigate | Active-invites table renders status/role/expiry only — never the raw token; the raw token is shown once via the generated-link card and only its hash is stored. |
| T-031-SET-5 | Information Disclosure | cross-tenant invite listing | mitigate | `listInvites(user.tenantId)` scopes to the caller's tenant; a missing tenantId yields an empty list (fail-closed). |
| T-031-SET-6 | Elevation of Privilege | missing gate on a new loader | mitigate | Both loader and action start with `requireUser`; the invite paths add `requireRole`/`can` — the capability matrix is the single source, no inline ad-hoc checks. |
</threat_model>
